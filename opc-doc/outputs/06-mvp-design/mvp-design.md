## 前置说明

**MVP 不是「简陋产品」，而是「最小验证」。**

MVP 的本质不是做一个功能不全的产品然后推向市场，而是用最小的成本和最短的时间，验证你最不确定的假设是否成立。一个有效的 MVP 实验，只需要回答一个问题：**这个假设是对的吗？**

如果假设被验证，我们再投入资源做完整版本；如果假设被推翻，我们及时转向，避免浪费。

本文档基于阶段01-04的已确认产出，尤其是阶段04识别的7个假设，设计MVP验证实验。

## 一、假设优先级排序

### 7 个假设全景

| 风险等级 | 假设                                           | 如果为假的后果              | 验证成本                   | 信息时效性                 |
| -------- | ---------------------------------------------- | --------------------------- | -------------------------- | -------------------------- |
| 🔴 极高  | H1：开发者愿为ADR内容付费                      | 收入模型崩塌，只能做纯开源  | 中（需要先有内容+流量）    | 需要流量和内容就位后才能测 |
| 🔴 极高  | H2：DSD搜索长尾流量真实存在且可达              | SEO渠道失效，须转向推式营销 | 低（发布几个页面即可观测） | 1个月内可得结果            |
| 🟠 高    | H3：DSD采用率会持续增长                        | 利基市场萎缩                | 近零（数据观测）           | 持续监控，非一次验证       |
| 🟠 高    | H4：一人能同时维持框架开发+内容产出+商业化运营 | 不可持续，质量下降          | 近零（时间记录）           | 需2-3个月才能判断          |
| 🟡 中    | H5：ADR格式作为内容形态对开发者有吸引力        | 需换内容载体                | 低（1-2篇文章+观测）       | 2-4周可得结果              |
| 🟢 低    | H6：商业化不与开源精神冲突                     | 被社区抵制                  | 近零                       | 自然验证                   |
| 🟢 低    | H7：中文DSD社区有足够付费意愿                  | 只能面向英文市场            | 低                         | 可与H1合并验证             |

### 优先级排序逻辑

**MVP 阶段最优先验证的 2 个假设：**

#### 🥇 优先验证 #1：H2 — DSD搜索长尾流量真实存在且可达

**为什么最优先**：

1. **下游一切依赖它**：没有流量，就没有人读到ADR，H5无从验证；没有人看到免费内容，H1付费意愿无从触发。H2是整条验证链的起点。
2. **验证成本最低**：只需发布5-7个ADR页面，等Google索引，观察Search Console数据。不需要写完整教程，不需要搭建付费系统。
3. **结果最快**：Google索引通常2-4周，1个月内可看到有机搜索信号。
4. **如果为假，转向成本最低**：如果SEO不可达，在投入大量内容生产之前就知道，避免沉没成本。

#### 🥈 优先验证 #2：H5 — ADR格式对开发者有吸引力

**为什么第二**：

1. **紧接H2之后**：如果有流量进来（H2通过），但没人读完ADR格式的文章，说明载体有问题——内容策略需要调整。
2. **验证成本低**：1-2篇ADR深度解读文章，发布到外部平台+自建站点，观察阅读完成率和互动。
3. **调整方向快**：如果ADR格式不受欢迎，可以快速换载体（视频/交互式/代码沙盒），而不影响已验证的流量渠道。

#### ⏳ 延后验证：H1 — 付费意愿

**为什么延后**：

- H1 是 H2 + H5 的下游假设。没有流量和内容吸引力，付费意愿无法被触发。
- 验证 H1 需要先有免费内容引流 + 付费产品着陆页，前置成本较高。
- 在 H2 和 H5 验证通过后，用 "Painted Door"（画门测试）低成本验证 H1。

#### 📡 持续监控：H3（市场增长）和 H4（一人可持续性）

- H3 不需要专门实验，每周花15分钟跟踪 DSD 生态指标即可
- H4 通过记录每周实际投入时间和产出来自然验证

#### 🔕 暂不验证：H6、H7

- H6 风险低，在MVP过程中自然验证（如果社区有负面反馈会立刻看到）
- H7 可与 H1 合并验证（付费产品同时支持中英文）

### 依赖关系与验证顺序

```
第1步：H2 验证（流量存在？）
  │
  ├── 通过 → 第2步：H5 验证（ADR格式吸引人？）
  │             │
  │             ├── 通过 → 第3步：H1 验证（付费意愿？）
  │             │             │
  │             │             ├── 通过 → 全面推进 🚀
  │             │             └── 失败 → Pivot收入模型 🔄
  │             │
  │             └── 失败 → Pivot内容载体 🔄
  │
  └── 失败 → Pivot获客渠道 或 Pivot利基 🔄

全程监控：H3（市场）+ H4（可持续性）
```

## 二、实验设计

### 实验 1：SEO 长尾流量探针（验证 H2）

#### 实验名称

**"5-7 ADR 页面上线，看Google是否送人来"**

#### 验证的假设

H2：DSD搜索长尾流量真实存在且可达

#### 最小验证形式

**不是**建完整的57页ADR网站——那太重了。最小验证形式是：

选择 5-7 个搜索意图最强、竞争最弱的 DSD 相关话题，各发布 1 个 ADR 页面到静态站点，等 Google 索引，观察 Search Console 的有机搜索数据。

**为什么 5-7 个而不是 1 个**：单个页面存在偶然性（可能刚好选错了关键词），5-7 个可以分散风险。但也不需要 57 个——我们不是做产品，而是在测信号。

**选页策略**：

- 从 57 份 ADR 中挑选与搜索意图最匹配的 5-7 份
- 优先选"开发者正在搜但找不到好答案"的话题
- 关键词方向："DSD SSR implementation"、"declarative shadow DOM production"、"DSD vs imperative shadow DOM"、"web components SSR without Lit"、"DSD hydration strategy"

#### 成功 / 失败标准

| 指标                          | 成功信号 | 失败信号 | 不确定信号 |
| ----------------------------- | -------- | -------- | ---------- |
| 月有机搜索访问量（5-7页合计） | ≥100     | <30      | 30-100     |
| 有页面进入Google搜索前3页     | ≥3个页面 | 0个页面  | 1-2个页面  |
| 平均点击率（CTR）             | ≥3%      | <1%      | 1-3%       |

**判定规则**：

- **通过**：月有机搜索 ≥100 且 ≥3个页面进入前3页 → DSD搜索长尾流量真实存在且可达
- **失败**：月有机搜索 <30 且 0个页面进入前3页 → DSD搜索长尾流量不可达或不存在
- **不确定**：中间地带 → 延长观察2周，优化页面SEO后复测

#### 所需时间和资源

| 项目                                  | 估算                      |
| ------------------------------------- | ------------------------- |
| ADR页面改写为SEO内容                  | 每篇2-3h × 5-7篇 = 10-21h |
| 静态站搭建（Vercel/Cloudflare Pages） | 3-5h                      |
| Google Search Console 配置            | 1h                        |
| 等待索引+数据收集                     | 3-4周                     |
| **总时间投入**                        | 约 15-27h（分布在3-5周）  |
| **金钱成本**                          | $0-15（域名费，托管免费） |

#### 实验执行步骤

1. **Week 1**：关键词研究（用 Google Trends、Search Console 现有数据、Ahrefs 免费工具），从57份ADR中选出搜索意图最匹配的5-7份
2. **Week 1-2**：将选出的ADR改写为SEO友好页面（每篇500-1500字，含标题/摘要/关键决策/代码示例），搭建静态站并发布
3. **Week 2**：提交 Google Search Console，手动请求索引；在掘金/Dev.to 发布1-2篇相关文章做外链
4. **Week 3-6**：等待索引，每周查看 Search Console 数据
5. **Week 6**：收集数据，判定实验结果

### 实验 2：ADR 格式吸引力测试（验证 H5）

#### 实验名称

**"1篇ADR深度解读，看开发者是否读完并互动"**

#### 验证的假设

H5：ADR格式作为内容形态对开发者有吸引力

#### 最小验证形式

**不是**写57篇ADR解读——那太重了。最小验证形式是：

选 1 份最具代表性的 ADR（涉及开发者最关心的决策：如"DSD-first vs SSR-first 渲染策略"），写一篇深度解读文章，同时发布到自建ADR站（免费版）和掘金/Dev.to（获客版），观察阅读完成率和互动。

**为什么 1 篇就够了**：如果 1 篇精心撰写的 ADR 解读都没人读完，那 57 篇也不会更好。1 篇足以测信号。

**文章结构**（模拟ADR格式）：

- Context：为什么面临这个决策
- Decision：我们做了什么选择
- Alternatives Considered：考虑了哪些替代方案
- Consequences：选择的结果和代价
- Code：可运行的代码示例

#### 成功 / 失败标准

| 指标             | 成功信号       | 失败信号 | 不确定信号 |
| ---------------- | -------------- | -------- | ---------- |
| 阅读完成率       | >40%           | <20%     | 20-40%     |
| 有意义评论数     | ≥5条（非水评） | 0条      | 1-4条      |
| 平均页面停留时间 | >3分钟         | <1分钟   | 1-3分钟    |
| 社交分享/转发    | ≥3次           | 0次      | 1-2次      |

**判定规则**：

- **通过**：阅读完成率 >40% 且 ≥5条有意义评论 → ADR格式有吸引力，继续
- **失败**：阅读完成率 <20% 且 0条评论 → ADR格式不被认可，需要换载体
- **不确定**：中间结果 → 再发1篇不同风格的ADR解读测试（是格式问题还是选题问题？）

#### 所需时间和资源

| 项目               | 估算                   |
| ------------------ | ---------------------- |
| 撰写1篇ADR深度解读 | 4-6h                   |
| 发布+配置分析      | 1-2h                   |
| 等待数据收集       | 2-3周                  |
| **总时间投入**     | 约 5-8h（分布在2-3周） |
| **金钱成本**       | $0                     |

#### 实验执行步骤

1. **Week 2**（与实验1同步）：选出1份最有故事性的ADR，改写为深度解读文章
2. **Week 2-3**：发布到自建ADR站 + 掘金/Dev.to，配置阅读分析（Google Analytics 事件追踪滚动深度）
3. **Week 3-5**：观察数据，回复评论，收集反馈
4. **Week 5**：判定实验结果

### 实验 3：付费意愿"画门测试"（验证 H1）

#### 实验名称

**"画一扇门，看有没有人推"**

#### 验证的假设

H1：开发者愿为ADR内容付费

> **"画门测试"解释**：在页面上放一个付费产品的入口（按钮/链接），点击后跳到一个说明"产品正在准备中，留下邮箱优先通知"的页面。如果没有人点，说明没人想要；如果有人点甚至留邮箱，说明付费意愿存在。这是成本最低的付费验证方式——你不需要真的做出产品就能测信号。

#### 最小验证形式

**不是**开发付费产品+支付系统——那太重了。最小验证形式是：

在实验1和实验2验证通过的前提下，在ADR免费页面和深度解读文章的末尾，添加一个CTA：

> "📖 这是ADR的精华版。完整的57份ADR深度解读包（含源码对照+架构Why）即将发布，[留下邮箱获取首发通知 + 早鸟折扣]"

点击后跳转到一个简单的邮箱收集页面（可用 Gumroad 预发布页 / Mailchimp 着陆页 / Carrd 单页）。

**关键**：CTA 文案要足够具体（说明产品是什么、为什么值），而不是泛泛的"订阅我们的通讯"。

#### 成功 / 失败标准

| 指标                                     | 成功信号 | 失败信号 | 不确定信号 |
| ---------------------------------------- | -------- | -------- | ---------- |
| CTA点击率（页面浏览→点击CTA）            | ≥5%      | <1%      | 1-5%       |
| 邮箱注册数（2周内）                      | ≥5人     | 0人      | 1-4人      |
| 主动询问（邮件/评论/私信问"什么时候出"） | ≥3人     | 0人      | 1-2人      |

**判定规则**：

- **通过**：≥5人留下邮箱 或 ≥3人主动询问完整版 → 付费意愿存在，值得开发付费产品
- **失败**：0人留邮箱 且 0人询问 → 付费意愿不足，需转向其他收入模型
- **不确定**：1-4人留邮箱 → 信号微弱，需要更多流量或更精准的CTA，延长观察期

#### 所需时间和资源

| 项目             | 估算                   |
| ---------------- | ---------------------- |
| 设计CTA+着陆页   | 2-3h                   |
| 配置邮箱收集工具 | 1h                     |
| 观察+跟进        | 持续2-3周              |
| **总时间投入**   | 约 3-4h（分布在2-3周） |
| **金钱成本**     | $0                     |

#### 实验执行步骤

1. **Week 5**（前提：H2和H5验证有正面信号）：在ADR站页面和掘金/Dev.to文章末尾添加CTA
2. **Week 5**：配置Gumroad预发布页或Carrd着陆页 + 邮箱收集
3. **Week 5-7**：观察点击率和注册数，通过邮件跟进注册者（了解他们最想要什么）
4. **Week 7**：判定实验结果

### 持续监控：H3 和 H4

#### H3 监控：DSD采用率会持续增长

**监控方式**（每周15分钟）：

- Google Trends："declarative shadow DOM"、"shadow DOM SSR" 趋势
- GitHub 搜索：DSD相关新repo数量变化
- npm 下载量：@webcomponents相关包、Lit下载量趋势
- Chrome Platform Status：DSD相关指标
- caniuse.com：DSD浏览器支持率变化

**预警信号**：

- 🟢 健康：DSD相关指标月增速 >5%
- 🟡 关注：增速放缓至 0-5%
- 🔴 警告：指标停滞或下降

#### H4 监控：一人可持续运营

**监控方式**（每周5分钟记录）：

- 本周实际投入时间（h）
- 本周实际产出（篇数/代码提交/互动数）
- 主观疲劳度（1-5分）

**预警信号**：

- 🟢 可持续：10-15h/周，≥2篇/月产出，疲劳度 ≤3
- 🟡 需调整：连续2周 >18h 或 <8h，产出 <1篇/月
- 🔴 不可持续：连续3周 >20h，或疲劳度持续4-5分

## 三、MVP 整体方案

### MVP 范围：做与不做

#### ✅ 做（MVP范围内）

| 组件                    | 目的                 | 最小形态                                 | 预估工时 |
| ----------------------- | -------------------- | ---------------------------------------- | -------- |
| ADR 微型网站            | 验证H2（搜索流量）   | 5-7个ADR页面 + 静态站 + Search Console   | 15-21h   |
| 1篇ADR深度解读          | 验证H5（格式吸引力） | 1篇长文，掘金+Dev.to+自建站同步发        | 5-8h     |
| 画门测试CTA             | 验证H1（付费意愿）   | 页面CTA + Gumroad/Carrd着陆页            | 3-4h     |
| LessJS Starter Template | B层获客验证          | 已有GitHub repo基础，补充README+一键运行 | 3-5h     |
| 分析工具配置            | 数据收集             | GA4 + Search Console + 简单事件追踪      | 2-3h     |

**MVP总工时**：约 28-41h

#### ❌ 不做（MVP范围外，验证通过后再做）

| 组件                               | 为什么现在不做                                | 什么时候做     |
| ---------------------------------- | --------------------------------------------- | -------------- |
| 完整57页ADR网站                    | 成本高，5-7页已足验证流量假设                 | H2通过后       |
| 3篇核心桥接教程                    | 1篇深度解读已足验证格式假设                   | H5通过后       |
| 付费产品开发（ADR包/Pro Template） | 先验证有人愿意买，再投入开发                  | H1通过后       |
| 支付系统接入                       | 画门测试不需要真实支付                        | H1通过后       |
| 邮件订阅系统                       | 简单邮箱收集足够                              | H1通过后       |
| YouTube视频内容                    | 制作成本高，MVP阶段非必要                     | H2+H5通过后    |
| 社群建设（Discord/论坛）           | 用户基数太小，社群无意义                      | 月访问量>500后 |
| 中文专属内容                       | 先验证英文SEO流量（搜索量更大），中文作为补充 | 英文验证通过后 |

### 与一人公司约束（10-15h/周）的兼容性分析

| 维度     | 评估        | 说明                                                                        |
| -------- | ----------- | --------------------------------------------------------------------------- |
| 周均工时 | ✅ 高度兼容 | MVP平均3.2h/周，远低于10-15h/周预算，留有充裕缓冲                           |
| 峰值周   | ✅ 可控     | Week 1-2 峰值约10-12h（搭建+改写），在预算范围内                            |
| 节奏适配 | ✅ 友好     | 创作集中在周末（改写ADR、写文章），数据观察可在工作日零散时间（15分钟）完成 |
| 并行风险 | ⚠️ 需注意   | 框架开发与MVP并行时，需明确时间分配（建议 7h框架 + 3hMVP）                  |
| 疲劳管理 | ✅ 低风险   | 数据等待期（Week 3-4, 5-6, 7-8）几乎无工作量，是天然休息期                  |

**结论**：MVP实验设计与10-15h/周约束高度兼容。最大优势是"等数据"的时间不需要投入工时，与框架开发自然交替。

### 各场景下的具体行动

#### 场景 A：三假设全部通过 ✅ → 全面推进

**条件**：H2 ≥100搜索/月 + H5 完读率>40% + H1 ≥5人留邮箱

**行动**：

1. 扩展ADR站至完整57页
2. 完成3篇核心桥接教程
3. 开发首个付费产品（优先ADR深度解读包，因为H1验证最直接）
4. 配置Gumroad支付
5. 开始构建邮件订阅体系
6. 预计 3-4 个月后进入稳定运营

#### 场景 B：H2通过，H5失败 → Pivot内容载体

**条件**：H2 ≥100搜索/月 + H5 完读率<20%

**含义**：流量存在但ADR格式不被认可

**行动**：

1. 保持ADR站（SEO占位价值仍在），但调整内容形态
2. 测试替代载体：
   - **代码沙盒式**：每份ADR配套一个StackBlitz/CodeSandbox可运行demo
   - **案例式**：从ADR提取真实bug/决策场景，写成"我踩了这个坑"的故事
   - **视频式**：5-10分钟短视频讲解核心决策
3. 重测H5'：新载体的阅读/观看完成率是否>40%
4. 核心洞察不变（57份ADR的内容价值），只是换了交付形式

#### 场景 C：H2失败，H3仍在增长 → Pivot获客渠道

**条件**：H2 <30搜索/月 + H3 DSD指标月增速>5%

**含义**：市场在增长但SEO不可达（可能关键词竞争大，或搜索意图不匹配）

**行动**：

1. 转向推式获客：
   - HN发布DSD技术文章（Show HN: LessJS）
   - Twitter/X技术圈互动（回复DSD相关讨论）
   - 掘金/InfoQ深度文+社群分发
   - Web Components社区Discord/Slack分享
2. 重测H2'：推式渠道能否带来月均100+有效访问
3. 如果推式渠道也失败 → 进入场景E

#### 场景 D：H2失败，H3停滞 → Pivot利基

**条件**：H2 <30搜索/月 + H3 DSD指标停滞或下降

**含义**：DSD利基市场可能不够大或在萎缩

**行动**：

1. 认真评估利基B「AI架构师方法论」（六维评分44/60，差距不大）
2. 用1-2周时间快速验证利基B的搜索流量假设
3. 利基B的优势：市场规模更大（AI话题）、搜索量更高
4. 利基A的产出（ADR格式经验、内容生产能力）可迁移到利基B

#### 场景 E：H1失败（H2+H5通过）→ Pivot收入模型

**条件**：H2通过 + H5通过 + H1 0人留邮箱

**含义**：有人看、有人读，但没人愿付费——内容有价值但变现模型错

**行动**：

1. 测试替代收入模型：
   - **赞助/打赏**：GitHub Sponsors + Buy Me a Coffee（低门槛）
   - **框架增值服务**：LessJS企业支持/咨询（高客单价）
   - **培训/工作坊**：线上DSD工作坊（$49-99/人，1对多）
   - **内容授权**：与企业合作内部培训材料
2. 重测H1'：新收入模型是否有需求信号
3. 考虑：如果内容本身无法直接变现，是否作为LessJS框架采用的获客工具（内容营销而非内容产品）

## 六、MVP阶段不做什么

明确列出**不做**的事，防止范围蔓延：

1. ❌ 不做完整57页ADR网站（5-7页足矣）
2. ❌ 不做3篇核心桥接教程（1篇深度解读足矣）
3. ❌ 不做付费产品开发（画门测试足矣）
4. ❌ 不做支付系统接入（画门不需要）
5. ❌ 不做邮件订阅系统（简单邮箱收集足矣）
6. ❌ 不做YouTube视频（制作成本高，延后）
7. ❌ 不做社群建设（用户基数不够）
8. ❌ 不做中文专属内容版本（先验证英文SEO）
9. ❌ 不做付费广告（ROI不明）
10. ❌ 不做品牌设计/Logo/视觉体系（内容质量>视觉包装）

**核心原则**：MVP阶段只做能验证假设的事，不做"感觉应该做"的事。

## 八、总结

### MVP设计的核心逻辑

**一句话**：用 5-7 个 ADR 页面测流量（H2），用 1 篇深度解读测格式（H5），用画门测试测付费（H1），3个实验串联，12周内得出 Go/No-Go 判断。

### 待验证的核心假设

1. **H2**：DSD搜索长尾流量真实存在且可达 → 用5-7页ADR微型站验证
2. **H5**：ADR格式对开发者有吸引力 → 用1篇深度解读文章验证

### MVP 形式

ADR微型站（5-7页）+ 1篇ADR深度解读 + 画门测试CTA

### 成功 / 失败判定标准

| 假设       | 成功标准                             | 失败标准                     |
| ---------- | ------------------------------------ | ---------------------------- |
| H2（流量） | 月有机搜索 ≥100，≥3页进入Google前3页 | 月有机搜索 <30，0页进入前3页 |
| H5（格式） | 完读率 >40%，≥5条有意义评论          | 完读率 <20%，0条评论         |
| H1（付费） | ≥5人留邮箱 或 ≥3人主动询问           | 0人留邮箱 且 0人询问         |

### 验证周期

12 周（3 个月），含数据等待和缓冲时间

### 与一人公司约束的兼容性

平均 3.2h/周，峰值 10-12h/周，在 10-15h/周预算内
