## 三、Pains（痛点）

### P1：知识断层——DSD 从 API 到生产之间没有桥

现有内容（web.dev、MDN、Chrome Blog）覆盖了「DSD 是什么、怎么用 API」，但**没有人讲「怎么在一个真实项目里用 DSD 做 SSR」**。搜索 "DSD SSR" 出来的要么是概念介绍，要么是 Lit 的黑盒方案。

> 开发者实际搜索路径：Google "declarative shadow DOM SSR" → 找到 web.dev 概念文章 → 不知道怎么用到自己项目 → 继续搜 → 找到 Lit SSR 方案 → 不用 Lit → 卡住

### P2：架构决策焦虑——没有「做还是不做」的判断框架

DSD 刚进 Baseline（2024 Q4），迁移窗口正打开，但：

- 不确定 DSD 的成熟度够不够上生产
- 不确定 DSD 和现有 SSR 框架能否共存
- 不确定哪些场景 DSD 有收益、哪些是过度工程
- 没有决策参考：别人是怎么判断的？考虑了哪些因素？

### P3：拼装地狱——DSD-SSR 渲染管线需要自己从零搭

没有一个现成的 DSD-first 渲染引擎。开发者必须自己拼：

- Node.js 环境下没有 `window` → 需要 DOM 模拟
- 每个组件手动写 `<template shadowrootmode="open">` → 可扩展性差
- SSR 输出的 HTML 和客户端 hydration 的对齐 → 需要自己设计协议
- 样式在 Shadow Root 内如何与 SSR 配合 → 没有最佳实践

### P4：兼容性地雷——Safari/Firefox 的实现差异

Safari 16.4+ 才支持 DSD，旧版本需要 polyfill。但：

- polyfill 和 SSR 的交互有坑
- 不同浏览器的 `<template shadowrootmode>` 解析行为不一致
- 没有完整的兼容性测试矩阵参考

### P5：孤立无援——DSD 社区太小，遇问题无人可问

- Stack Overflow 上 DSD 相关问题极少，高质量回答更少
- 没有像 React/Vue 那样的成熟社区可以求助
- 尝试 DSD 的开发者感觉自己「在荒野中独行」

### P6：内容信任危机——现有 DSD 教程质量参差

- 中文社区有大量 DSD 介绍文章，但大多是 web.dev 的翻译/改写，没有实战深度
- 英文社区同样缺乏「real world」案例
- 重复的入门内容充斥搜索结果，深度内容缺失

## 五、价值主张陈述

### 3 个候选方向

#### 方向 A：决策先行型

> **对于** 正在评估 DSD 是否适合自己项目的前端架构师和独立开发者，
> **他们需要** 在 DSD 技术选型中做出有信心的架构决策，
> **我们提供** 基于 57 份真实架构决策记录（ADR）的 DSD 实战决策框架 + 可运行的参考实现，
> **不同于** web.dev 的 API 文档和 Lit 的黑盒 SSR 方案，
> **我们的核心优势是** 全球唯一的 DSD-first 框架生产级经验——我们不是在讲 DSD 的 API，而是在讲「我们为什么这样用了 DSD」。

**关键词**：决策信心、ADR、生产级经验、Why > How

#### 方向 B：路径打通型

> **对于** 想用 Web Components + DSD 做 SSR 但找不到完整实现路径的开发者，
> **他们需要** 一条从零搭建 DSD-SSR 渲染管线的可验证实战路径，
> **我们提供** 基于 LessJS 框架的端到端 DSD-SSR 教程 + 可 fork 的项目模板，
> **不同于** 零散的博客教程和需要自己拼装的碎片化方案，
> **我们的核心优势是** 唯一已经跑通 DSD-first 全链路（渲染→hydration→样式隔离→构建打包）的开源框架。

**关键词**：可验证路径、全链路、开源框架、端到端

#### 方向 C：深度独特型

> **对于** 需要深度理解 DSD 与 SSR 架构本质的前端架构师和框架贡献者，
> **他们需要** 穿透 API 表层，理解 DSD 设计的架构 trade-off 和浏览器实现差异，
> **我们提供** 57 份架构决策记录 + DSD-first 引擎源码解析 + 浏览器兼容性实测矩阵，
> **不同于** 浅层 API 教程和翻译式中文内容，
> **我们的核心优势是** 从 0.1 版本到 0.28 版本持续迭代 DSD-first 的 3 年架构决策沉淀——这是任何其他内容来源都没有的。

**关键词**：架构深度、3 年沉淀、trade-off 解析、差异化洞察

### 初步判断

- **方向 A 是最佳主价值主张**：它是 B 和 C 的上游——开发者先做决策（A），再需要路径（B），深理解（C）是加分项
- **方向 B 是最自然的入口内容**：搜索流量最大，是获客漏斗的顶部
- **方向 C 是信任建设器**：小众但高影响力，建立「这个人真的懂」的专家认知

**建议组合**：以 A 为价值主张核心承诺，以 B 为内容获客入口，以 C 为信任加固层。

### 7.2 搜索体验还原

开发者搜索 "DSD SSR" 时的典型路径：

```
搜索 "declarative shadow DOM SSR"
  → 找到 web.dev 的 DSD 介绍文章（2024年）
    → 知道了 DSD 是什么，但不知道怎么用
  → 搜索 "DSD SSR tutorial"
    → 找到 Lit SSR 方案
      → 不用 Lit / 不想绑 Lit → 卡住
  → 搜索 "web components SSR without framework"
    → 找到 dev.to 的概念讨论文章
      → 好文但没代码 → 卡住
  → 搜索 "declarative shadow DOM production"
    → 几乎没有结果
      → 彻底卡住 ❌
```

**空白本质**：从「知道 DSD」到「用 DSD 做出东西」之间，存在一个**没有桥的鸿沟**。

### 7.4 关键洞察

1. **搜索长尾特性**：DSD 不是流量型话题，而是搜索型长尾——搜索者带着明确问题来，需要明确答案。这恰好匹配 ADR 的格式：每个决策 = 一个搜索问题。

2. **内容真空 = SEO 机会**：搜索 "DSD SSR production" 或 "DSD architecture decision" 几乎没有高质量结果。57 份 ADR 中每个都是一篇潜在的 SEO 占位内容。

3. **可信度壁垒**：即使有人想模仿，3 年的 ADR 沉淀和 DSD-first 框架代码是时间壁垒，无法快速复制。
