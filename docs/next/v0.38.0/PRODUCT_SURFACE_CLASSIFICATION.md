# v0.38.0 Product Surface Classification

This document classifies the current package and export surface recorded in
`docs/next/v0.38.0/PACKAGE_SURFACE_INVENTORY.md`.

Classifications are operational for docs and template cleanup. New package names
or intentional removals still require ADR approval before implementation.

## Classification Rules

| Classification     | Meaning                                                                         |
| ------------------ | ------------------------------------------------------------------------------- |
| `product`          | Primary user-facing surface that root docs and starter templates may recommend. |
| `advanced-subpath` | Public and supported, but documented only in advanced or reference material.    |
| `internal`         | Implementation detail. Remove from public tutorial/recommended paths.           |
| `archived`         | Kept for compatibility or evidence, but not a v1 product direction.             |
| `removed`          | Scheduled for removal with migration notes and ADR coverage.                    |
| `adr-required`     | Cannot be finalized until an ADR approves a name or removal decision.           |

## Package Name Decisions

| Candidate                | Status         | Reason                                                                                             |
| ------------------------ | -------------- | -------------------------------------------------------------------------------------------------- |
| `@openelement/elements`  | `adr-required` | Could clarify the Web Component authoring product, but overlaps `core` and `runtime`.              |
| `@openelement/protocol`  | `adr-required` | Could simplify singular naming, but current package is published as `protocols`.                   |
| `@openelement/framework` | `adr-required` | Could bundle app, routing, SSG/SSR/ISR, and build ownership, but would be a new top-level package. |

No candidate name is approved for implementation in v0.38.0 until an ADR exists.

## Required Follow-Up

- Root READMEs should stop presenting `@openelement/hub`, `@openelement/ssg`,
  and raw `@openelement/adapter-vite` as primary product packages.
- `@openelement/create` should hide protocol subpaths and adapter-vite build
  ownership from generated starter imports unless those remain intentionally
  advanced public APIs.
- Architecture docs should update the package count from the older 19-package
  wording and align layer labels with this classification.
- Migration notes are required before any `internal`, `archived`, or `removed`
  surface is physically removed.
