# LessJS v0.23.0 - Layered Package Architecture

**2026-05-27** | v0.23.0 | ADR-0050, SOP v0.23.0, package graph, runtime facade, protocols

v0.23.0 closes the architecture-first release line introduced by ADR-0050.
This release does not try to add more full-stack surface. It makes the existing
framework easier to publish, explain, and extend by putting shared contracts,
runtime authoring, configuration assembly, and package graph validation in the
right layers.

The practical release question was:

> Can LessJS keep `@lessjs/core` as a small DSD runtime kernel while generated
> projects, feature packages, adapters, and JSR publish order all agree on the
> same dependency graph?

The answer is now yes. All packages are aligned to `0.23.0`, generated projects
use the intended authoring facade, and `deno task graph:check` verifies the
package graph instead of relying on local workspace resolution.

---

## Release Line Summary

### Protocols Own Shared Build Contracts

`@lessjs/protocols` is now the owner for shared build context contracts and
virtual module ids.

This removes the old architecture smell where feature packages needed
adapter-owned paths just to participate in build-time integration.

Canonical paths:

- `@lessjs/protocols/build-types`
- `@lessjs/protocols/virtual-ids`

`@lessjs/content`, `@lessjs/i18n`, and `@lessjs/adapter-vite` now consume these
contracts from the protocols layer.

Reference: [ADR-0050](../adr/ADR-0050-layered-package-architecture.md)

### Runtime Is The Authoring Facade

`@lessjs/runtime` is the authoring surface for components.

Generated routes and islands now import authoring APIs such as `DsdElement`,
`html`, `StyleSheet`, and signal helpers from `@lessjs/runtime`. This keeps
ordinary app code ergonomic without forcing `@lessjs/core` to become the
product-wide DX barrel.

`@lessjs/core` stays focused on the runtime kernel:

- `DsdElement`
- template rendering
- `renderDSD`
- island runtime contracts
- navigation
- logger and runtime errors

### App Stays The Configuration Facade

`@lessjs/app` remains the `lessjs()` configuration facade for Vite projects.

It composes the Vite adapter, content, and i18n integration, but no longer
imports `@lessjs/adapter-vite/build-context`. That makes the application facade
a public configuration boundary instead of a consumer of adapter internals.

### Signals Stay An alien-signals Facade

LessJS does not own the low-level reactivity algorithm. `@lessjs/signals` is the
LessJS facade over `alien-signals`.

The framework owns the public signal contract and DSD integration semantics.
The engine remains isolated below that facade.

### Package Graph Is Now A Release Gate

`deno task graph:check` now verifies the real package graph:

- all LessJS packages use one version line;
- package cycles are rejected;
- publish order is checked against dependencies;
- every package in `packages/` must be present in the publish workflow;
- source-level direct `@lessjs/*` imports must be declared in that package's
  local `deno.json`.

This catches mistakes that the monorepo root import map can otherwise hide.

### Publish Workflow Covers All Packages

The JSR publish workflow now includes all 18 packages, including
`@lessjs/runtime`.

This matters because `@lessjs/create` generated projects import
`@lessjs/runtime`; publishing create without publishing runtime in the same
version line would create a broken consumer graph.

## User-facing Outcome

New generated projects teach the intended architecture:

```ts
import { DsdElement, html, signal, StyleSheet } from '@lessjs/runtime';
```

Configuration remains separate:

```ts
import { lessjs } from '@lessjs/app';
```

The generated `deno.json` declares direct imports instead of relying on
transitive or monorepo-only resolution.

---

## Breaking Changes

The release intentionally removes wrong-owner paths instead of preserving them
as compatibility bridges.

- Shared build contracts are no longer owned by adapter-vite public paths.
  Use `@lessjs/protocols/build-types`.
- Virtual module ids are no longer owned by adapter-vite public paths.
  Use `@lessjs/protocols/virtual-ids`.
- Ordinary component authoring should use `@lessjs/runtime`, not
  `@lessjs/core` as a product-level convenience barrel.
- Removed stale root aliases and adapter-vite exports for old contract
  locations.

This is deliberate. v0.23.0 is an architecture cleanup line, not a
backward-compatibility line.

The release was validated with:

```sh
deno task graph:check
deno task lint
deno task typecheck
deno fmt --check
deno task docs:check-strategy
deno test packages/create/__tests__/cli.test.ts --allow-read --allow-write --allow-env --allow-net --allow-run --allow-ffi

```

The full site build passes. Existing Shoelace SSR/client-only diagnostics and
bundle budget warnings remain visible during the build, but they do not block
the v0.23.0 architecture release.

## Bug Fixes (post-architecture)

### E2E Theme System Fix

All 6 theme-system E2E tests were timing out because `<less-theme-toggle>` lives
inside `<less-layout>`'s shadow DOM. `document.querySelector('less-theme-toggle')`
returned `null` in all `page.evaluate()` calls, causing `waitForToggleReady` to
hit the 30s timeout every time.

Fix: added `findToggleInPage()` helper that searches through
`less-layout.shadowRoot` first, falling back to light DOM. All `page.evaluate()`
calls updated.

Result: 86 passed, 6 failed → **92 passed, 0 failed**.

## Deferred

v0.23.0 deliberately does not ship:

- ISR production handlers;
- CF Workers KV or Deno KV adapters;
- generic auth, ORM, database, session, or Node server abstractions;
- a v1.0 API freeze.

Those belong to v0.24+ and later work after the package graph stays clean.

---

## Release Note Short Form

v0.23.0 is the Layered Package Architecture release. It introduces
`@lessjs/protocols` as the shared contracts owner, confirms `@lessjs/runtime`
as the component-authoring facade, keeps `@lessjs/app` focused on configuration,
and strengthens the package graph gate so JSR publish order, direct imports, and
unified versions are mechanically checked.

For users, the important change is simple: write components from
`@lessjs/runtime`, configure projects through `@lessjs/app`, and rely on the
0.23.0 package set as one coherent release line.
