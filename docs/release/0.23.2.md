## Root Cause Analysis (consumer-smoke)

The smoke test creates a consumer project via `@openelement/create`, then runs
`deno task build` (SSG pipeline). Three stacked bugs prevented this:

### 1. `runtime` missing from SSG package resolver whitelist

`packages/adapter-vite/src/ssg-package-resolver.ts` maintains a
`DEFAULT_LESSJS_PACKAGES` whitelist for resolving `@openelement/*` JSR
packages during consumer SSG builds. `runtime` was not in this list.

**Fix**: Added `'runtime'` to both `DEFAULT_LESSJS_PACKAGES` and
`LESSJS_EXPORT_FILES`.

### 2. Version mismatch on JSR

The SSG resolver uses adapter-vite's own version to resolve ALL
`@openelement/*` packages. When only adapter-vite was bumped to 0.23.1,
it tried to fetch `@openelement/runtime/0.23.1/` etc. â†’ JSR 404.

**Fix**: Bumped all 17 packages to 0.23.2 (unified version).

### 3. `alien-signals` not in consumer import map

`@openelement/signals` depends on `alien-signals` (npm). The SSG resolver's
`rewriteNpmSpecifiers` converts `npm:alien-signals@3.2.1` back to bare
`alien-signals`. Rolldown bundles it into the SSR entry, but when
`buildSSG()` runs `import()` to execute it, Deno cannot resolve
`alien-signals` â€” it's not in the consumer project's import map.

**Fix**: Added `"alien-signals": "npm:alien-signals@^3.2.0"` to the
consumer template's `deno.json` imports in `packages/create/cli.ts`.

## Maintenance Rules

1. **New `@openelement/*` package**: Must also update `ssg-package-resolver.ts` whitelist
2. **Version bump**: Always bump ALL 17 packages together (resolver uses unified version)
3. **npm dependency with `rewriteNpmSpecifiers`**: If exposed at the consumer SSR
   boundary, add to template's import map
