# LessJS v0.21.16 - JSR Consumer Hardening Complete

**2026-05-25** | v0.21.5 to v0.21.16 | JSR consumer build, SSG pipeline, and CI release gates

v0.21.16 closes the post-v0.21.5 hardening line. After v0.21.5 completed the
pre-v0.22 gate, the following patch train focused on one practical release
question: can a user create a LessJS project from JSR, build it outside the
monorepo, and rely on a clean Deno/JSR/Vite module graph?

The answer is now yes. LessJS owns package graph normalization and runtime
polyfills; Vite/Rolldown owns final bundling, tree-shaking, and code splitting.
The CI model now separates source validation from post-publish JSR validation
and Windows compatibility monitoring.

---

## Release Line Summary

### v0.21.5 - Pre-v0.22 Gate

v0.21.5 completed the v0.21.x hardening gate before v0.22 work could start.
The release classified the public Core API, validated DSD conformance, proved
the generated SSG starter, tightened Hub trust checks, and aligned status and
roadmap documentation with real command results.

The v0.22 scope was deliberately narrowed to edge handlers, API route mounting,
static assets, ISR cache adapters, deployment examples, and showcase proof. Auth,
ORM, sessions, generic permissions, and full RPC remained out of scope.

Reference: [SOP-010](../sop/v0.21.x/SOP-010-v022-entry-gate.md)

### v0.21.6 - Create CLI JSR Metadata

`@openelement/create` was updated to use the current JSR metadata endpoint:
`https://jsr.io/@openelement/<pkg>/meta.json`.

The CLI now reads the `latest` field instead of relying on stale metadata shapes,
so generated projects resolve the intended LessJS package set.

### v0.21.7 - Core Subpath Resolution

adapter-vite fixed SSG resolution for `@openelement/core/*` source files fetched from
JSR.

The resolver appends `.ts` where needed and keeps relative imports inside the
virtual LessJS package namespace instead of leaking unresolved JSR details into
Vite/Rolldown.

### v0.21.8 - UI Package Resolution

The core-only resolver became a LessJS package resolver.

This added support for published UI subpaths such as `@openelement/ui/less-card`,
allowing SSR bundle generation to consume LessJS UI modules from JSR instead of
depending on monorepo-only paths.

### v0.21.9 - ESM Module Graph First

v0.21.9 formalized the long-term consumer build boundary: LessJS owns the
Deno/JSR package graph, and Vite/Rolldown owns final JavaScript optimization.

The centralized resolver covers bare LessJS package specifiers and
`jsr:@openelement/*` specifiers, including `@openelement/signals/framework`,
`@openelement/content/sitemap`, `@openelement/i18n`, and published UI subpaths.

Reference: [SOP-011](../sop/v0.21.x/SOP-011-jsr-consumer-esm-graph.md)

### v0.21.10 - Consumer Smoke Gate Design

The release introduced consumer smoke coverage, then exposed the publish-order
problem: a pre-publish CI job that runs `jsr:@openelement/create` tests the previously
published package set, not the source branch being validated.

That lesson led to the later split between source-backed CI, post-publish smoke,
and scheduled JSR latest monitoring.

References:
[SOP-012](../sop/v0.21.x/SOP-012-ci-consumer-smoke.md),
[SOP-013](../sop/v0.21.x/SOP-013-post-publish-consumer-smoke.md)

### v0.21.11 - npm Specifier Rewrite

adapter-vite rewrote `npm:` specifiers found in JSR source modules into bare
specifiers Vite/Rolldown can resolve in the generated consumer project.

This reduced unresolved import failures caused by Deno-style source graphs being
handed directly to a Node-style bundler.

### v0.21.12 - SSR Runtime and External Dependency Resolution

v0.21.12 fixed multiple SSR/SSG runtime issues:

- `customElements` became a Map-backed polyfill instead of a no-op registry.
- `@openelement/core` became self-contained for SSR by owning its HTMLElement stub.
- external dependencies such as `parse5`, `entities`, and `hono` moved to Deno
  pre-resolution, preventing subpath leaks such as `entities/lib/escape.js`.

This release made the SSR runtime safer and reduced brittle manual external
lists.

References:
[SOP-016](../sop/v0.21.x/SOP-016-ssr-htmlelement-self-contained.md),
[SOP-017](../sop/v0.21.x/SOP-017-deno-pre-resolution-external-deps.md)

### v0.21.13 - Remote URL Guard

JSR remote execution exposed a `fileURLToPath()` assumption in client build
logic.

v0.21.13 guarded remote execution paths so HTTPS module URLs from JSR are not
treated as local `file:` URLs.

### v0.21.14 - Phase 2 Package Island Resolver

Phase 2 client island builds now reuse the LessJS JSR package resolver.

This fixed consumer builds where `virtual:less-client-entry` could not resolve
package islands such as `@openelement/ui/less-card`.

### v0.21.15 - Content Nav Defaults

`content.nav.routesDir` now defaults to `app/routes`, and the create template
writes the same value explicitly.

This fixed consumer builds where `path.resolve()` received `undefined` during
content navigation scanning.

### v0.21.16 - No-op Outer Build Trigger

The outer Vite build no longer bundles the generated Hono SSR entry as a browser
artifact.

Instead, LessJS uses a no-op virtual build trigger for the outer build while the
real SSR and client island builds run in their dedicated phases. v0.21.16 also
fixed remote `@openelement/core/*` URL derivation so core source files are resolved
from `@openelement/core`, not from `@openelement/adapter-vite`.

Reference: [SOP-018](../sop/v0.21.x/SOP-018-ci-release-gate-separation.md)

## CI and Release Gate Model

v0.21.16 separates three different validation jobs that had previously been
mixed together:

| Layer                       | Workflow                                     | Purpose                                                                                                                    |
| --------------------------- | -------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Source-backed SOP Gate      | `.github/workflows/sop-gate.yml`             | Validate the current branch with formatting, lint, typecheck, audit, tests, build, DSD, Hub, docs, starter proof, and e2e. |
| Post-publish Consumer Smoke | `.github/workflows/publish.yml`              | Validate the freshly published JSR package set on Ubuntu after publish completes.                                          |
| Windows JSR Latest Monitor  | `.github/workflows/jsr-consumer-monitor.yml` | Monitor `jsr:@openelement/create` on Windows by schedule or manual dispatch without blocking source branches.              |

This avoids the misleading failure mode where branch CI tests an older JSR
release, while still preserving real consumer validation after publishing.

Reference:
[ADR-0048](../adr/ADR-0048-ci-release-gate-separation.md)

## User-facing Outcome

A generated LessJS project can now be created from JSR and built outside the
monorepo with a clean ownership boundary:

```sh
deno run -A jsr:@openelement/create test-blog
cd test-blog
deno task build
```

LessJS handles JSR package discovery, virtual source modules, SSR-safe runtime
polyfills, and generated entrypoints. Vite/Rolldown handles production bundling.

## Release Note Short Form

v0.21.16 completes the v0.21.x JSR consumer hardening line. The patch train from
v0.21.5 to v0.21.16 turns Reactive DSD into a release-hardened consumer stack:
generated projects build outside the monorepo, LessJS owns Deno/JSR package
graph normalization, SSR/DSD runtime polyfills are self-contained, external
dependency resolution is deterministic, and CI now separates source gates from
post-publish JSR validation.
