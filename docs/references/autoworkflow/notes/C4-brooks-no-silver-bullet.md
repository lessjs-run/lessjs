# C4. Brooks：没有银弹

> Brooks, F. P. "No Silver Bullet: Essence and Accidents of Software Engineering." _IEEE Computer_, 1987.
> （Turing Award 1999 的主要贡献之一）

## 核心论点

Brooks 区分了软件工程的两种困难：

| 类型                      | 定义                       | 例子                                | 能消灭吗 |
| ------------------------- | -------------------------- | ----------------------------------- | -------- |
| **本质困难 (Essential)**  | 软件本身固有的复杂性       | 需求模糊、状态空间巨大、系统间交互  | ❌ 不能  |
| **偶然困难 (Accidental)** | 我们使用的工具和方法带来的 | 手动管理内存、没有类型检查、没有 CI | ✅ 能    |

**银弹** = 一种能一举消灭所有软件工程困难的技术或方法。

Brooks 的结论：没有银弹。本质困难是永恒的。

## 为什么这跟我们的自演化系统相关

### AutoFlow 消灭的是偶然困难

| 偶然困难           | AutoFlow 怎么消灭   |
| ------------------ | ------------------- |
| 手动 bump 版本号   | Cell 自动做         |
| 忘了更新 changelog | 从 SOP 证据自动生成 |
| 版本号 dep 不一致  | graph:check 拦住    |
| 发布的包有类型错误 | typecheck 拦住      |
| 忘了跑某一道门禁   | Harness 自动全跑    |

这些都是**偶然困难**——不是软件本身复杂，是我们的流程容易出错。AutoFlow 自动化的是机器人该干的活。

### AutoFlow 消灭不了本质困难

| 本质困难                                | 为什么消灭不了       |
| --------------------------------------- | -------------------- |
| 这个 API 设计是对的吗？                 | 判断需要人的领域知识 |
| 这个 breaking change 的迁移成本值得吗？ | 需要人对用户的理解   |
| 该不该在这个时候发版本？                | 需要人的战略判断     |
| 这个 ADR 的 trade-off 合理吗？          | 需要人的架构经验     |

AutoFlow 不会替你回答这些问题。它只是确保：**回答了这些问题的证据都在仓库里。**

## 那 AI 呢？AI 能消灭本质困难吗？

Brooks 在 1995 年的论文再版后记中专门回应了：

> AI 不会消灭本质困难。但 AI 可以让人更高效地处理本质困难。

这不是悲观。是现实主义。

我们的架构承认这个现实：**AI 处理偶然困难（自动化），人处理本质困难（决策）。** 两者各司其职。

## 对应到 Cell 类型

| Cell          | 消灭的困难类型                     |
| ------------- | ---------------------------------- |
| version-bump  | 偶然——机械操作                     |
| changelog     | 偶然——从证据生成                   |
| fmt-fix       | 偶然——格式规范                     |
| code-refactor | 偶然 + 微量的本质——有 harness 约束 |
| api-change    | 本质——必须人审                     |
| write-adr     | 本质——必须人写                     |

**Cell 的 risk 分类 = 偶然困难 vs 本质困难的边界。**
