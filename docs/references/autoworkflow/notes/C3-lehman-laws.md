# C3. Lehman 软件演化定律

> Lehman, M. M. "Programs, Life Cycles, and Laws of Software Evolution." _Proceedings of the IEEE_, 1980.
> Lehman, M. M., and Belady, L. A. _Program Evolution: Processes of Software Change._ Academic Press, 1985.

## 来处

1970 年代，Lehman 在 IBM 研究了 OS/360 的演化历史——几十个版本，十几年时间。他问：

> 软件随时间怎么变？有没有**规律**，像物理定律一样？

他找到了 8 条。不是数学定理——是来自大量真实数据的经验法则。

## 八条定律中跟我们有直接关系的四条

### 第一定律：持续变化 (Continuing Change)

> 一个用在实际环境中的软件系统，必须持续改变，否则就会变得越来越不令人满意。

为什么？因为软件的环境（硬件、用户需求、竞争产品、法律法规）在变。软件不跟着变，就过时了。

**对 openElement**：这不是消极观测。这是我们系统的**存在理由**。如果一个框架不持续演变（bump 版本、更新文档、修复依赖），它就会被遗忘。AutoFlow 就是实现持续变化的引擎。

### 第二定律：复杂度增长 (Increasing Complexity)

> 随着系统演化，其复杂度会增加——除非有专门的工作来减少或维持它。

每次改动都会增加交互：新模块打乱了旧模块的假设，新 API 多了调用路径，新配置多了组合爆炸。

**对 openElement**：这不只适用于代码，也适用于 governance 文档。v0.30 CONTRACT.md 是一页，v0.34 STATUS.md 已经 170 行。版本越多，文档越长，内部引用越密。**复杂度的敌人是裁撤。** v0.38 Public Surface Reset 就是专门为这个留的版本——剪掉死包、统一路径、让复杂度降回去。

### 第六定律：持续增长 (Continuing Growth)

> 系统的功能内容必须持续增长，以维持用户满意度。

不是 bug fix。是**新功能**。如果不加新东西，用户会觉得"这个项目死了"。

**对 openElement**：这也是 v0.36（rendering）+ v0.37（server/data/UI）存在的理由。AutoFlow 不是目的——它是**让做产品功能更安全**的手段。没有产品功能的框架没有用户，没有用户的框架不需要演化。

### 第七条：质量下降 (Declining Quality)

> 除非严格维护并适配运行环境的变更，一个演化中的软件系统的质量会持续下降。

本质是**熵**。测试会过时，文档会漂移，类型会变松。

**对 openElement**：Harness 就是对抗熵的机制。fmt/lint/typecheck/test 不是一次性的——它们是**持续运行的熵检测器**。Lehman 没说"质量会下降，别管了"，他说的是"你必须持续投入才能对抗下降"。Harness 就是这个持续投入的自动化形式。

## 对自演化系统的核心洞察

Lehman 定律揭示了一个深层矛盾：

```
自演化 → 改代码 → 复杂度 ↑ → 质量 ↓ → 需要更多 Harness
         ↑________________________________↓
                   自演化又生成新的复杂性
```

这意味着：**自演化系统如果不治理自己，会加速自己的熵增。** 它改得越快，烂得越快。

我们的答案是三层防御：

1. **Harness**（截住烂代码）
2. **v0.38 大剪枝**（周期性降低复杂度）
3. **Governance 文档**（保持"知道自己在做什么"的能力）
