# v0.15.1 — 2026-05-16

## Summary

v0.15.1 is a hardening release for the v0.15 renderer kernel. It closes the
audit gates required before v0.16 package protocol work starts: safe package
island metadata handling, stricter generated client code, safer PWA caching, and
non-skipping build artifact checks.

## Changes

- **Security**: `generateClientEntry()` now validates island metadata and emits
  tag names and module specifiers with `JSON.stringify()` instead of template
  interpolation.
- **Security**: Package island metadata is validated before entering generated
  client code. Invalid `tagName`, unsafe `modulePath`, control characters,
  line breaks, and unsupported protocols now fail fast.
- **Registry boundary**: `scanPackageIslands()` documents the current dynamic
  `import(pkg).islands` path as trusted backward compatibility only. Future
  third-party registry work should use static manifests.
- **PWA safety**: Generated service workers no longer cache non-GET requests,
  authorization-bearing requests, `/api/`, `/rpc/`, auth/session/login/logout
  paths, or ordinary HTML/API responses.
- **Build safety**: `www/__tests__/build-output.test.ts` now checks `www/dist`
  directly and fails when required build output is missing.
- **Tests**: Added audit-gate coverage for malicious package island metadata,
  hardened service worker rules, Hono virtual entry cleanup, and
  `assertExists(boolean)` regressions.
- **create CLI tests**: Template version assertions now validate resolved
  `${v.*}` placeholders instead of stale hard-coded version fragments.
- **Versioning**: All workspace package versions and internal `@openelement/*`
  constraints are bumped to `0.15.1`.

## Known Issues

- Full `deno task fmt:check` is blocked by existing formatting diffs in v0.16
  planning documents, not by the v0.15.1 source/test changes.
- The legacy dynamic package island discovery path still executes explicitly
  configured package entries. It remains supported for compatibility, but v0.16
  should move registry discovery to static manifests.

## Breaking Changes

- No public API breaking changes.

# v0.15.1 — 2026-05-16（中文）

## 摘要

v0.15.1 是 v0.15 renderer kernel 的加固版本。它先补齐进入 v0.16 package
protocol 前必须关闭的审计门槛：package island 元数据安全、生成客户端代码安全、PWA
缓存边界、以及不会静默跳过的构建产物测试。

## 变更

- **安全**：`generateClientEntry()` 现在会校验 island metadata，并用
  `JSON.stringify()` 生成 tag 名和 module specifier，不再用模板字符串直接拼接。
- **安全**：package island metadata 在进入生成代码前会被校验。非法
  `tagName`、危险 `modulePath`、控制字符、换行和不支持的协议会直接失败。
- **Registry 边界**：`scanPackageIslands()` 明确当前动态
  `import(pkg).islands` 只是可信兼容路径。未来第三方 registry 应走静态
  manifest。
- **PWA 安全**：生成的 service worker 不再缓存非 GET 请求、带
  Authorization 的请求、`/api/`、`/rpc/`、auth/session/login/logout 路径，也不再缓存普通
  HTML/API 响应。
- **构建安全**：`www/__tests__/build-output.test.ts` 改为直接检查
  `www/dist`，构建产物缺失时会失败，不再静默跳过。
- **测试**：新增恶意 package island metadata、service worker 缓存边界、Hono
  virtual entry 清理、`assertExists(boolean)` 回归的审计门槛测试。
- **create CLI 测试**：模板版本断言改为检查 `${v.*}` 占位符，不再依赖过期硬编码版本片段。
- **版本**：所有 workspace package 版本和内部 `@openelement/*` 约束升级到
  `0.15.1`。

## 已知问题

- 完整 `deno task fmt:check` 仍被 v0.16 规划文档中的既有格式差异挡住，不是
  v0.15.1 源码或测试改动导致。
- 旧的动态 package island discovery 仍会执行显式配置的 package entry。该路径继续作为兼容能力保留，但
  v0.16 应迁移到静态 manifest discovery。

## 破坏性变更

- 无公开 API 破坏性变更。
