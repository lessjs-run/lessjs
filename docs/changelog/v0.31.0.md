# Changelog - v0.31.0

Date: 2026-06-05

## Summary

v0.31.0 introduces the JSX-first Application API. App authors now start from
`definePage()` and `defineIsland()` instead of extending `DsdElement` directly.

## Added

- **Application API**: `@openelement/app` now exports `definePage()`,
  `defineIsland()`, `defineElement()`, and `defineLayout()`.
- **Page descriptors**: `definePage()` supports function shorthand and an object
  form with `title`, `description`, `head`, `layout`, `styles`, `rendering`,
  `streaming`, `revalidate`, `load`, and `render`.
- **Route data wiring**: adapter-vite generated SSR and SSG entries read
  `default.openElementPage`, call `load()`, and pass data/request/params through
  internal `__openElement*` props.
- **Vite subpath**: `@openelement/app/vite` now owns the `openElement()` Vite
  configuration facade.
- **Docs planning**: roadmap now maps v0.32-v0.36 and v1.0, with dedicated SOP
  folders for each future minor.
- **ADR**: ADR-0082 documents the Application API split and the `/vite`
  configuration boundary.

## Changed

- **Starter DX**: generated projects now use `definePage()`, `defineIsland()`,
  and `@openelement/app/vite`.
- **Current docs**: README, package READMEs, current architecture docs, status,
  and www guide pages teach Application API first.
- **Internal SSR props**: internal route data/request/params are filtered from
  DSD host attributes and `data-ssr-props`.
- **Consumer smoke tooling**: local consumer smoke now respects the new `/vite`
  import and injects aliases without rewriting the public entry to adapter
  internals.

## Breaking Changes

- `openElement()` is no longer imported from `@openelement/app`.

```diff
- import { openElement } from '@openelement/app';
+ import { openElement } from '@openelement/app/vite';
```

The package root is now reserved for application authoring helpers.

## Known Issues

- `DsdElement` remains public as a runtime primitive and is still used internally
  by existing www pages. This is intentional; the new default tutorial path is
  Application API first, not a forced migration of every internal route file.
