# v0.21.x - 2026-05-24

## Summary

v0.21.x after v0.21.5 is the JSR consumer hardening line. The release train
keeps the Reactive DSD architecture stable while fixing the generated-project
path so a user can create a LessJS app from JSR, build it outside the monorepo,
and let Vite/Rolldown perform the final bundle, tree-shaking, and code
splitting over a clean ESM module graph.

## v0.21.6

### Create CLI JSR Metadata Fix

- `@lessjs/create` switched to the current JSR metadata endpoint:
  `https://jsr.io/@lessjs/<pkg>/meta.json`.
- Remote create mode reads the JSR `latest` field instead of the stale
  `latestVersion`/array fallback shape.
- The release line was aligned to `0.21.6` because JSR packages are immutable
  and generated projects must resolve the fixed create CLI and package set.

## v0.21.9

### ESM Module Graph First Consumer Build

- ADR-0041 and SOP-011 define the long-term boundary: LessJS owns the ESM
  module graph and package entrypoint contract; Vite/Rolldown owns final
  bundling, tree-shaking, and code splitting.
- adapter-vite now delegates JSR consumer package graph normalization to a
  centralized SSG package resolver instead of keeping one-off logic in
  `build-ssg.ts`.
- The resolver understands bare LessJS package specifiers and
  `jsr:@lessjs/*` specifiers, including `@lessjs/signals/framework`,
  `@lessjs/content/sitemap`, `@lessjs/i18n`, and published UI subpaths.
- Relative imports from virtual JSR source modules remain inside the virtual
  LessJS package namespace before Vite sees the graph.
- All packages are aligned to `0.21.9`; internal LessJS `jsr:` imports use the
  `^0.21.9` release train instead of broad `^0.21` ranges.

### Consumer Gate

After publishing the full `0.21.9` package set, validate the public path outside
the monorepo:

```sh
deno run -A jsr:@lessjs/create test-blog
cd test-blog
deno task build
```

The build must not emit unresolved `jsr:@lessjs/*` imports from Vite/Rolldown.

## Publishing Note

JSR packages are immutable. A failed published package cannot be corrected in
place, so `0.21.9` must be published as a complete package set:

```sh
deno task publish
```

Do not treat the release as complete until the generated consumer smoke passes
outside the LessJS workspace.

## v0.21.12

### Fixed

- **SSR DSD rendering crash**: `customElements` polyfill used no-op `define()`/`get()`, causing `renderDSDByName()` to return void elements for all components. Fixed with Map-backed implementation. ([SOP-016](../sop/v0.21.x/SOP-016-ssr-htmlelement-self-contained.md))

- **HTMLElement polyfill regression**: Bare `class HTMLElement {}` in `output.banner` caused all components to lose DOM methods. Moved HTMLElement stub to `@lessjs/core/dsd-element.ts` — `@lessjs/core` is now self-contained with `_SsrHTMLElementStub` (6 members: `hasAttribute`, `getAttribute`, `setAttribute`, `removeAttribute`, `tagName`, `isConnected`). ([SOP-016](../sop/v0.21.x/SOP-016-ssr-htmlelement-self-contained.md))

- **ESM subpath leak into Rolldown**: `entities/lib/escape.js` subpath imports from `parse5` could not be resolved by bare-string external lists. Fixed with Deno pre-resolution ([ADR-0047](../adr/ADR-0047-deno-pre-resolution-external-deps.md)).

### Changed

- `packages/core/src/dsd-element.ts`: `_HTMLElement` fallback replaced with `_SsrHTMLElementStub`, assigned to `globalThis.HTMLElement`
- `packages/adapter-vite/src/cli/build-ssg.ts`: `output.banner` — HTMLElement removed, customElements uses `Map`; `ssr.external` auto-generated via `external-resolver.ts`
- `packages/adapter-vite/src/external-resolver.ts`: New — Deno pre-resolution core module
- `packages/content/deno.json`: Added `section-matter` transitive dependency
