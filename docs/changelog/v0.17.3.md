# v0.17.3 - Multi-Framework Adapters

**Release date**: 2026-05-16

## Summary

Adds two new SSR adapter packages expanding LessJS beyond Lit:
`@lessjs/adapter-vanilla` for plain Web Components and
`@lessjs/adapter-react` for React-to-Web-Component bridging.

This release also records an important compatibility boundary: multi-adapter
support does not mean arbitrary third-party UI libraries are automatically
SSR-safe.

Final documentation pass: v0.17.3 is now closed as the adapter expansion and
compatibility-boundary release. Enforcement work moves to v0.17.4.

## Changes

- **@lessjs/adapter-vanilla**: New package for plain/vanilla Web Components
  whose `render()` returns a string directly.
  - `installVanillaAdapter()`: registers the `vanilla` adapter for DSD rendering
  - `extractVanillaStyles()`: extracts static `styles` property from vanilla
    component classes
  - `DsdVanillaElement`: base class extending `HTMLElement` with DSD detection,
    hydration event binding, and AbortController cleanup
  - `WithDsdHydration()`: mixin for custom base class composition
  - zero framework dependencies

- **@lessjs/adapter-react**: New package for React components wrapped as Web
  Components.
  - `installReactAdapter()`: registers the `react` adapter for DSD rendering
  - `isReactElement()`: detects React elements via `$$typeof`
  - `renderReactToString()`: converts React elements to HTML strings via
    `ReactDOMServer.renderToStaticMarkup()`
  - `DsdReactElement`: base class with `getReactElement()` for SSR rendering and
    client-side `createRoot()` mounting
  - peer deps: `react@^19`, `react-dom@^19`

- **Workspace config**: Both packages added to root `deno.json` workspace,
  imports, typecheck, and publish tasks.

- **Tests**: 26 new tests. Total release gate at the time of tagging:
  554 unit tests + 92 e2e passing.

- **Docs**: ADR/SOP/roadmap now use a conservative third-party package
  admission model. A package must have an explicit supported renderer or LessJS
  SSR declaration before it is treated as SSR-capable.

- **Release planning**: STATUS and SOP index now split v0.17.4 -> v1.0 into
  executable version steps with entry criteria, verification, exit criteria,
  and non-goals.

## Compatibility Boundary

v0.17.3 supports multiple declared renderer families. It does not guarantee
that packages such as Shoelace or Media Chrome can be SSR-rendered merely
because their custom elements are registered.

Browser-dependent libraries should be handled as client-only until v0.18+
validation proves a stronger capability. This avoids late build errors from
components that require real DOM APIs during SSR.

## Exit Status

- v0.17.3 documentation is complete.
- v0.17.3 does not claim universal third-party SSR.
- v0.17.4 is the next required patch before v0.18: enforce client-only imports
  before SSR bundle generation.

## Breaking Changes

None. This is a minor release: two new packages and documentation changes, no
intentional breaking change to existing packages.

## Known Issues

- 3 JSR `unanalyzable-dynamic-import` warnings in adapter-vite are expected for
  runtime-only dependencies and are not publish blockers.
- `DsdVanillaElement` and `DsdReactElement` use Proxy-based lazy initializers to
  avoid `ReferenceError` in non-DOM Deno test environments.
- Browser-dependent third-party UI libraries may fail if imported into the SSR
  bundle before their `ssr: false` or compatibility metadata is applied. This
  is now tracked as the v0.17.4 hardening gate.
