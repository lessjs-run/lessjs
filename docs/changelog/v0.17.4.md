# v0.17.4 - Compatibility Boundary Hardening

> **Release type**: patch / correctness and build robustness
> **Date**: 2026-05-16

## Summary

v0.17.4 turns the v0.17.3 compatibility boundary into enforceable build
behavior: client-only and browser-dependent Web Components no longer enter the
SSR path by accident, and one-command builds now exit cleanly. A post-tag fix
round resolved dev-mode SSR crashes caused by island modules evaluating in the
SSR runner before the DOM shim provides `HTMLElement`.

## Changes

### SSR Admission

- Added an `SsrAdmissionPlan` to the Vite entry descriptor so SSR entry
  generation is driven by explicit renderable/client-only decisions.
- Local islands can now expose static `less` metadata such as `ssr: false`,
  `dsd`, and `hydrate` without being imported during route discovery.
- Package islands are treated conservatively before v0.18 validation: they are
  client-only unless future manifest validation explicitly admits them for SSR.
- Nested custom-element rendering skips client-only tags instead of
  instantiating browser-only child components on the server.

### Browser-Only Package Handling

- Package manifest scanning now skips browser-only packages that fail with
  missing browser globals such as `window`, instead of failing the entire build.
- The docs site can include Shoelace/Media Chrome style browser-first demos
  without producing repeated SSR stack traces.

### Optional Adapter Resolution

- Added optional stubs for `@lessjs/adapter-lit`,
  `@lessjs/adapter-vanilla`, and `@lessjs/adapter-react` in both the primary
  Vite plugin and the SSG sub-build path.
- `create-less` generated projects can build through the one-command pipeline
  even when optional adapters are not installed locally.

### Build Exit And Budgets

- The one-command build CLI now exits explicitly after successful Vite
  completion, fixing the post-build hang observed in v0.17.3 work.
- The docs build-output test now separates core client budget from showcase
  demo budget so demo UI libraries do not hide core bundle regressions.

### Dev-Mode SSR Crash Fixes

These fixes address runtime errors that appeared after the initial v0.17.4 tag
when running `deno task dev`:

- **SSR-safe base class pattern**: `WithDsdHydration(globalThis.HTMLElement)`
  crashes in the SSR module runner because `HTMLElement` is `undefined` at
  module evaluation time (the DOM shim has not loaded yet). Fixed in
  `react-showcase.ts` and `media-chrome-showcase.ts` with a conditional:
  `typeof globalThis.HTMLElement !== 'undefined' ? WithDsdHydration(...) : class {}`.
  When `HTMLElement` is unavailable, the island falls back to a plain class so
  module evaluation succeeds; browser behavior is unaffected since `HTMLElement`
  is always available there.

- **ADR 0014 patch now applies in dev + SSG**: The `customElements.define()`
  idempotent patch in `entry-renderer.ts` was guarded by `if (desc.isSSG)`,
  so dev-mode SSR got duplicate-define crashes. Removed the guard — the patch
  now applies unconditionally because island modules call `customElements.define()`
  as a side-effect in both modes, and the SSR dom-shim is not idempotent.

- **media-chrome dynamic import**: Changed `import 'media-chrome'` (static,
  evaluates browser-only code at module load) to a dynamic `import('media-chrome')`
  inside `connectedCallback()`, guarded by `typeof globalThis.HTMLElement !==
  'undefined'`. Media Chrome registers custom elements that require DOM APIs at
  import time; deferring to `connectedCallback` ensures the import only runs in
  the browser.

- **Dev task working directory**: `deno task dev` was running from the project
  root, but the route scanner uses `process.cwd()` for relative paths. Changed
  the task to `cd www && deno run ...` so `process.cwd()` matches the Vite root.
  Also added `--allow-ffi --allow-sys` flags required by Vite 8 rolldown native
  bindings.

- **api-consumer fetch timing**: Deferred the status fetch to
  `this.updateComplete.then()` in `connectedCallback()`, avoiding Lit update
  races inside a parent DSD shadow root.

- **WithDsdHydration connectedCallback infinite recursion**: The
  `WithDsdHydration` mixin in both `adapter-vanilla` and `adapter-react` used
  `Object.getPrototypeOf(Object.getPrototypeOf(this))` to call the parent
  class's `connectedCallback`. When the subclass (e.g. `ReactShowcase`) does
  not override `connectedCallback`, the two-step prototype walk finds the
  mixin's own method, causing infinite recursion (`Maximum call stack size
  exceeded`). Fixed by using the closure-captured `superClass` variable
  directly: `superClass.prototype.connectedCallback.call(this)`. This affects
  all components using `WithDsdHydration(HTMLElement)` as their base class.

### Reporting And Tests

- `dsd-report.json` now includes admission decisions from the SSR plan.
- Added tests for local metadata scanning, admission decisions, client-only
  nested tags, optional package stubs, and build-output budget boundaries.

## Second-Round Findings (Post-Tag, 2026-05-16)

After the initial v0.17.4 tag, a deployment-time investigation into a reported
`Failed to fetch` SW error and header/footer style loss uncovered three
independent issues that were fixed in subsequent commits.

### SW Cache Robustness

The Service Worker error `sw.js:19 Uncaught (in promise) TypeError: Failed to
fetch` had two causes:

- **Navigate requests (HTML pages)**: The SW used bare `fetch(e.request)` with
  no catch handler. When the network was unreachable (e.g. an expired Cloudflare
  Pages preview URL such as `aad0fe18.lessjs.pages.dev`), the uncaught promise
  rejection produced the `sw.js:19` TypeError.
- **Asset requests**: The `cacheFirst` strategy used `fetch()` without a
  try/catch. Network failures on asset fetches also propagated as uncaught
  rejections.

**Fix** (`packages/adapter-vite/src/cli/ssg-render.ts`):

1. Navigate requests now use `networkFirst` with an offline fallback (empty 408
   response) instead of bare `fetch(e.request)`.
2. Asset requests in `cacheFirst` wrap `fetch()` in try/catch, returning a 408
   response on network failure instead of throwing.
3. Non-asset, non-navigate requests (e.g. opaque third-party requests) are
   passed through without SW interception.

### Package Island SSR Admission

**Root cause**: `buildSsrAdmissionPlan()` in `entry-descriptor.ts` treated _all_
`source === 'package'` islands as client-only, regardless of their declared
`ssr` metadata:

```ts
// Old behaviour — package islands were always client-only
else if (source === 'package') {
  renderPath = 'client-only';
  reason = island.ssr === true
    ? 'package SSR import is deferred until v0.18 admission validation'
    : 'package island has no validated SSR capability';
}
```

This meant that even when a package island like `@lessjs/ui/less-layout`
declared `ssr: true` in its manifest, it was excluded from SSR. The
`less-layout` component — which wraps the entire docs-site header/footer — had
no DSD template in the SSG output. Its styles lived entirely in the Lit Shadow
DOM, meaning any JS failure prevented the component from upgrading, producing a
styling collapse of the entire page chrome.

**Systemic issue**: The conservative "package islands deferred until v0.18"
decision was sound as a _startup policy_ but became a correctness problem when
well-validated first-party packages (`@lessjs/ui`, `@lessjs/app`) declared
explicit SSR capability. The policy needed to trust its own manifest metadata
for packages that _do_ have validated SSR capability.

**Fix** (`packages/adapter-vite/src/entry-descriptor.ts`):

Package islands with `ssr: true` in their manifest now take the `ssr+client`
render path instead of being forced to `client-only`. The conservative default
(`client-only`) still applies to packages without explicit `ssr: true`
declaration.

### Nested DSD Rendering

**Root cause**: `render-nested.ts` had a function `isInsideDsdTemplate()` that
skipped any custom element appearing inside a `<template shadowrootmode="open">`
tag. The assumption was that such content was "already rendered output" and
should not be processed again. However:

- `less-layout` appeared inside `docs-home`'s DSD shadow DOM (as light DOM /
  slot content)
- `less-layout` had **no DSD template of its own** — it was just a bare
  `<less-layout>` placeholder tag
- `isInsideDsdTemplate()` saw the parent DSD template and skipped it, so
  `less-layout` never got its own DSD output

This is valid nested DSD: a component's shadow DOM can legally contain another
component with its own DSD template.

**Fix** (`packages/core/src/render-nested.ts`):

Deleted the `isInsideDsdTemplate()` check entirely. The existing
`elementAlreadyHasDSD()` check already correctly handles the case where a
component _already_ has a DSD template (from the parent's `render()` output),
so the extra guard was redundant and harmful.

### Effect

- `less-layout` now renders a full DSD template in SSG output (header, nav,
  theme-toggle, footer, and their styles are present in declarative shadow DOM)
- Every SSG page gained ~4 additional DSD templates (from ~9 to ~13), including
  `less-layout`, `less-theme-toggle` (inside layout), and other nested
  components
- The `@lessjs/ui` package manifest's `ssr: true` metadata is now respected
- SW no longer crashes on transient network failures
- 554 tests passing

### Third-Round Findings (2026-05-17)

Two additional fixes applied after confirming the deployed site.

#### Vanilla Adapter `render()` Fallback

**Root cause**: `WithDsdHydration(HTMLElement)` mixin in
`@lessjs/adapter-vanilla` only handled DSD hydration (detecting pre-populated
shadow roots and binding events). It had no mechanism to call `render()` when
no DSD content existed. This affected any vanilla-adapter island with
`ssr: false` — the shadow root was created empty and `render()` was never
invoked.

Compare with other adapters:

| Adapter | Base class                      | ssr   | Render mechanism                           | Result  |
| ------- | ------------------------------- | ----- | ------------------------------------------ | ------- |
| Lit     | `LitElement`                    | false | Lit lifecycle auto-calls `render()` ✅     | Visible |
| React   | `WithDsdHydration(HTMLElement)` | true  | DSD content from SSR → direct hydration ✅ | Visible |
| Vanilla | `WithDsdHydration(HTMLElement)` | false | No DSD content + no auto-render ❌         | Empty   |

`media-chrome-showcase` was the first component to hit this — its
`render()` returned the full Media Chrome template, but nothing ever
populated the shadow root with it.

**Fix** (`packages/adapter-vanilla/src/dsd-hydration.ts`):

Added a client-side render fallback to the mixin's `connectedCallback()`:
when `_dsdHydrated` is false (no pre-existing DSD content) and `this.shadowRoot`
exists, it checks for a `render()` method on the instance and calls it:

```ts
if (this.shadowRoot) {
  const renderFn = (this as Record<string, unknown>).render;
  if (typeof renderFn === 'function') {
    const html = renderFn.call(this);
    this.shadowRoot.innerHTML = String(html);
  }
}
```

This goes in the mixin itself, not as a workaround in each island — all
vanilla adapter consumers benefit automatically.

### Effect

- Vanilla adapter `ssr: false` islands now render on first paint
- The fix is in the mixin layer, not per-island — any future vanilla
  adapter island with `ssr: false` works automatically
- 554 tests passing

### Consumer-Level Fixes (2026-05-17)

After deploying the v0.17.4 third-round mixin fix, two issues remained
visible on the deployed dev page (`https://dev.lessjs.pages.dev/`):

**1. Shoelace styles missing** — `shoelace-showcase.ts` imported component
JS (`@shoelace-style/shoelace/dist/components/...`) but not the theme
CSS. Without `light.css`, Shoelace components rendered as unstyled
bare elements, producing the "everything looks wrong" effect.

Analogy: buying IKEA furniture (components) but skipping the hardware
pack (CSS theme) — the parts are there but nothing looks or behaves right.

**2. Media Chrome empty** — `media-chrome-showcase.ts` never created a
shadow root. The `WithDsdHydration` mixin's `connectedCallback()` only
calls `render()` when `this.shadowRoot` exists. Since no shadow root
was ever attached, the render output was never injected and the card
stayed blank.

Analogy: the house frame (custom element) is built, but nobody opened
the door to let the decoration crew (render function) inside.

**Fixes**:

- `shoelace-showcase.ts`: Added
  `import '@shoelace-style/shoelace/dist/themes/light.css'` so the
  Shoelace theme styles load alongside the components.

- `media-chrome-showcase.ts`:
  - Added `this.attachShadow({ mode: 'open' })` in `connectedCallback()`
    before `super.connectedCallback()`, so the mixin detects the shadow
    root and invokes `render()`.
  - Moved `static styles` into `render()` output as an inline `<style>`
    tag, since the vanilla adapter does not auto-apply `static styles`.

**Effect**:

- Shoelace buttons, inputs, and alerts now display with correct theme
  styles on the deployed page.
- Media Chrome player controls (play, timeline, volume, fullscreen)
  now render correctly in the showcase section.
- The mixin-level fix (third round) + these consumer-level fixes together
  produce correct rendering for all three showcase islands.

## Known Issues

- The docs demo bundle still reports budget warnings because it intentionally
  includes Media Chrome, React, and Shoelace showcase chunks. This is tracked as
  demo weight, not a core engine regression.
- Browser-only packages skipped during manifest discovery are not validated
  packages yet. v0.18 must replace this conservative skip with a CEM/manifest
  classifier.
- The conditional base class pattern (`typeof HTMLElement !== 'undefined' ?
  WithDsdHydration(HTMLElement) : class {}`) is a local workaround in showcase
  islands. A framework-level solution should be provided by the adapter packages
  so every island author does not need to repeat this guard.
- Package islands rely on side-effect `customElements.define()` from module
  imports. v0.18 should formalise package island SSR registration in the
  admission plan rather than relying on side effects.

## Breaking Changes

- None for public runtime APIs.
