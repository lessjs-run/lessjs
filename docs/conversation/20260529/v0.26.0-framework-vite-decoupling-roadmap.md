## 1. 为什么现在要做这件事

### 1.1 触发事件

v0.25.0 刚刚完成全部 14 个 SOP 发布。这是我们连续多个版本做"架构债务清算"的最后一站。AD-0049 将 v0.22 从 Edge Full-Stack 重置为 Architecture Integrity，ADR-0050 确立了 Layered Package Architecture，v0.23/v0.24/v0.25 一路做 package boundary repair、adapter decomposition、declarative DX。

在这些清理工作完成后，一个根本性问题浮现了：**框架核心与 Vite 构建工具的耦合**。

### 1.2 问题本质

LessJS 的定位是"Web Standards-first, DSD-first, static-first, low-magic, deterministic"。但当前实现中，有三类 Vite 虚拟模块（virtual modules）深度嵌入在路由代码和框架生成代码中：

| 虚拟模块                 | 影响范围                               | 耦合类型                      |
| ------------------------ | -------------------------------------- | ----------------------------- |
| `virtual:less-nav`       | 55 个路由文件                          | 路由层 → Vite 虚拟模块        |
| `virtual:less-blog-data` | 5 个路由文件 + entry-renderer 生成代码 | 路由层/框架层 → Vite 虚拟模块 |
| `virtual:less-i18n-data` | entry-renderer 生成代码                | 框架层 → Vite 虚拟模块        |

这意味着：框架层的路由组件和生成的 entry 代码**不能脱离 Vite 虚拟模块机制独立运行**。这与我们"dev 模式零 bundler（Deno serve 直接跑源码）"的愿景矛盾。

## 2. 架构审计发现（摘要）

### 2.1 虚拟模块的由来

`virtual:less-nav` 出现在 ADR-0018 时期。当时的设计决策是：路由组件不应从 `@openelement/content` 模块级状态读取数据，而应从虚拟模块获取，保证"零模块状态（zero module state）"。

这个设计在当时是正确的——它解决了模块级状态导致的 HMR 不一致和 SSR/CSR 数据不同步问题。但今天来看，**虚拟模块作为"零状态数据的传输通道"可以被更简单的机制替代：框架包子路径导出**。

### 2.2 耦合的具体形式

```
当前数据流：
  content/i18n package → buildStart() → ctx.plugins.xxxOptions → virtual module plugin → virtual:less-xxx → route component

理想数据流：
  content/i18n package → 子路径导出 → @openelement/content/nav, @openelement/i18n/data → route component
```

Vite 虚拟模块在当前数据流中扮演的是"数据管道"角色——它本身不产生数据，只是把 `ctx.plugins.xxxOptions` 转换为 ESM 导出。这个角色完全可以被框架包的普通 ESM 导出替代。

### 2.3 受影响的文件清单

- **`virtual:less-nav`**：55 个路由文件（`www/app/routes/**/*.ts` 和 `.tsx`）
- **`virtual:less-blog-data`**：5 个路由文件 + `packages/adapter-vite/src/entry-renderer.ts` 第 599 行
- **`virtual:less-i18n-data`**：`packages/adapter-vite/src/entry-renderer.ts` 第 604 行
- **对应的虚拟模块插件**：`packages/content/src/index.ts`（virtualNavPlugin）、`packages/content/src/blog-data-plugin.ts`、`packages/i18n/src/i18n-data-plugin.ts`

## 3. 理想架构图

```
┌─────────────────────────────────────────────────────────┐
│                    框架层（纯 ESM）                       │
│                                                         │
│  @openelement/core        @openelement/content       @openelement/i18n │
│  (runtime kernel)    ├─ /nav               ├─ /data     │
│                      ├─ /blog              ├─ /routes   │
│                      └─ /sitemap           └─ ...       │
│                                                         │
│  ↓ 子路径导出（纯 ESM，零 Vite 依赖）                     │
│                                                         │
│  路由组件 import { navSections } from '@openelement/content/nav' │
│  路由组件 import { posts, getPostBySlug } from '@openelement/content/blog' │
│  路由组件 import { locales } from '@openelement/i18n/data'     │
│                                                         │
│  ↓ 生产构建时                                            │
│                                                         │
│                构建适配器层（Vite Adapter）                │
│  @openelement/adapter-vite: alias 映射 + tree-shaking         │
│  virtual:less-nav → '@openelement/content/nav'（仅 alias）    │
│                                                         │
│  Vite 插件不做数据注入，只做打包与优化                      │
└─────────────────────────────────────────────────────────┘
```

### 关键原则

1. **框架层零构建工具依赖**：`@openelement/content` 和 `@openelement/i18n` 不应依赖 Vite 虚拟模块机制来暴露数据
2. **数据层框架承包导出**：导航数据、博客数据、i18n 数据通过包的子路径导出，与 Deno import map、JSR 导出完全兼容
3. **构建适配器只做打包**：Vite 插件的职责收缩为 alias 映射 + tree-shaking + code splitting，不做数据注入

## 4. 各方讨论视角

### 4.1 产品经理（许清楚）视角

**关注点**：用户迁移成本、API 稳定性、版本兼容性。

55 个路由文件需要批量修改 import 路径。这是一个"一次性迁移成本"。关键是：

- 迁移必须可以自动化（写一个 codemod 脚本）
- `virtual:less-nav` → `@openelement/content/nav` 的语义等价性必须保证
- 迁移后的 API 应该比 `virtual:less-xxx` 更直观——用户不再需要理解"虚拟模块"是什么

**建议**：v0.26.0 应该同时提供：

- 新的子路径导出 API（推荐使用）
- 一个版本的 `virtual:less-xxx` 别名兼容层（deprecated），给用户一个版本的缓冲期

### 4.2 架构师（高见远）视角

**关注点**：包依赖图、构建顺序、dev 模式行为、SSR 数据一致性。

当前 `virtual:less-nav` 之所以存在，是因为 ADR-0018 要求"零模块状态"。虚拟模块的 `load()` 钩子在每次 SSR 请求时重新执行，保证数据新鲜。如果换成 `@openelement/content/nav` 子路径导出：

- **模块缓存问题**：Deno/Vite 的 ESM 模块缓存可能使导航数据在 dev 模式下不更新。解决方案：`@openelement/content/nav` 导出的是函数而非静态对象，或者利用 Deno 的 `--reload` 和 Vite 的 HMR invalidate 机制。
- **SSR 一致性**：子路径导出在 SSR 和 CSR 环境下加载的是同一个 ESM 模块。这比虚拟模块（仅在 Vite 环境下存在）更可预测。
- **包依赖图改善**：`@openelement/content` 当前依赖 `@openelement/protocols/virtual-ids`。移除虚拟模块后，content 包的依赖会更干净。

**建议**：先做 `@openelement/content` 和 `@openelement/i18n` 的子路径导出，确认 dev/build 两端行为一致，再做路由文件迁移。

### 4.3 用户视角

**关注点**：我的代码需要改什么？改了之后有什么好处？

当前用户（www 站点自身就是 LessJS 的 dogfooding 用户）在路由文件中写：

```typescript
import { navSections } from 'virtual:less-nav';
import { posts } from 'virtual:less-blog-data';
import { locales } from 'virtual:less-i18n-data';
```

迁移后：

```typescript
import { navSections } from '@openelement/content/nav';
import { getPostBySlug, posts } from '@openelement/content/blog';
import { getDefaultLocale, locales } from '@openelement/i18n/data';
```

好处：

- IDE 可以跳转到定义（虚拟模块无法做到）
- TypeScript 类型检查正常工作（虚拟模块需要 `declare module`）
- 可以在任何 ESM 环境 import（不限于 Vite）
- 概念更简单：不需要理解"虚拟模块"

## 5. 最终路线图

经过三方讨论，达成以下共识：

### v0.26.0：Framework-Vite Decoupling

| 阶段    | 任务                                                                                           | 优先级 |
| ------- | ---------------------------------------------------------------------------------------------- | ------ |
| Phase 1 | `@openelement/content` 新增 `/nav`、`/blog` 子路径导出                                              | P0     |
| Phase 1 | `@openelement/i18n` 新增 `/data` 子路径导出                                                         | P0     |
| Phase 2 | 更新 55 个路由文件：`virtual:less-nav` → `@openelement/content/nav`                                 | P0     |
| Phase 2 | 更新 5 个路由文件：`virtual:less-blog-data` → `@openelement/content/blog`                           | P0     |
| Phase 2 | 更新 entry-renderer 生成代码：移除 `virtual:less-blog-data` 和 `virtual:less-i18n-data` 的生成 | P0     |
| Phase 3 | 移除 content/i18n 中的虚拟模块插件代码                                                         | P1     |
| Phase 3 | 移除 adapter-vite 中的虚拟模块 resolveId/load 逻辑                                             | P1     |
| Phase 4 | Vite 插件只保留 alias 映射（virtual → framework 包路径），不做数据注入                         | P1     |
| Phase 5 | 验证 dev/build/SSG/SSR/E2E 全流程                                                              | P0     |

### 向后兼容策略

- v0.26.0：新 API（子路径导出）为主，旧 API（虚拟模块）保留但标记 deprecated，alias 兼容
- v0.27.0：移除虚拟模块兼容层
- 迁移脚本作为 `@openelement/create` 或 deno task 的一部分提供

### 不在此版本的范围

- 完全移除 Vite（生产构建仍然使用 Vite 做打包）
- Deno serve 直接跑源码（需要 Deno 生态进一步成熟后再评估）
- 多 bundler adapter（v1.0 之后再考虑）

## 6. 风险与缓解

| 风险                                                               | 缓解措施                                               |
| ------------------------------------------------------------------ | ------------------------------------------------------ |
| 55 个文件的批量 import 迁移容易出错                                | 编写 codemod 脚本自动化，CI 中 typecheck + test 验证   |
| 子路径导出的模块缓存导致 dev 模式数据不刷新                        | 使用函数导出（惰性求值）代替静态对象导出               |
| `@openelement/content/blog` 导出与 `@openelement/content` 主入口产生循环依赖 | blog 子路径使用独立的入口文件，不经过主 index.ts       |
| 生成的 entry 代码中移除虚拟模块导致 SSR 构建中断                   | 先在 entry-renderer 中做 feature flag 切换，渐进式迁移 |
