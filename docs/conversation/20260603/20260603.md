## 1. CF Pages Deployment Fix

### Problem

Cloudflare Pages failed during automatic Git builds with:

```text
NotCapable: Requires env access to "CI"
```

### Root Cause

Vite's CJS dependency chain includes `picocolors`, which reads `process.env.CI`
through Deno's Node compatibility layer. In the CF Pages build environment this
triggered Deno's permission checks. Wrapping the build in a custom script made
the permission boundary harder to reason about because child commands received a
new permission context.

### Fix

- Use the repo's `deno task build` directly in CF Pages.
- Avoid an extra `Deno.Command` wrapper for the deployment build path.
- Install Deno in the CF Pages command before running the build:

```sh
curl -fsSL https://deno.land/install.sh | sh && export PATH="$HOME/.deno/bin:$PATH" && deno task build
```

### Lesson

After changing build files, push before re-testing CF Pages. Otherwise the
remote build is still testing the previous commit.

## 2. LessJS UI Shell Direction

### Core Problem

The LessJS DSD Shell layer does not yet have a CSS-only component library. The
current `@lessjs/ui` components are JSX + `DsdElement` components, so they are
better treated as interactive Island components rather than zero-JS Shell
building blocks.

### Architecture Direction

LessJS should split UI into two tracks:

```text
@lessjs/ui/css
  Ocean Shell components
  CSS-only, DSD-friendly, zero client JS
  tokens + base + component CSS

@lessjs/ui
  Island components
  JSX + DsdElement
  client hydration when interaction requires JavaScript
```

The intended model is:

- Ocean components cover static structure and native HTML interaction.
- Island components cover focus management, keyboard behavior, positioning,
  state, network calls, and other JavaScript behavior.
- Both tracks share the same token and theme system.

### Candidate Strategy

The selected planning direction is a carefully scoped daisyUI 5 CSS fork:

- start from compiled CSS, not Tailwind plugin internals
- adapt selectors for DSD and Shadow DOM
- move theme variables to a root-level CSS variable bridge
- keep component CSS inside the relevant DSD template when isolation matters
- use Open Props as the base token layer

This is not a small patch. It belongs to `v0.29.0`, after `v0.28.6` finishes the
build and metadata cleanup line.
