# v0.25.0 → v0.26.0: 声明式与响应式路线图

> 日期: 2026-05-29 | 架构师: 齐活林 | 基于 v0.24.3 codebase 实际情况

---

## 架构师诚实评估

先列事实，再定路线。

### 当前框架实际使用情况

```
static props = { ... }       ← 在 10 个 UI 组件中用
signal()                      ← 在 counter-island + reactive-showcase 中用
computed()                    ← 0 处生产使用
effect()                      ← 仅 dsd-element.ts 内部 VNode tracking
_middleware.ts                ← 0 个文件存在 (scanner/descriptor 支持但无人用)
content system (blog/ADR)     ← 已处理数据获取 (markdown → virtual module → SSG)
static data 声明式 fetch      ← 不存在，当前 content system 已覆盖此场景
```

**核心结论**：框架的声明式基座（`static props`、文件路由）已好。响应式基础设施（signal/computed/effect）已就绪但使用率低。不要在无人使用的功能上盖楼。

**前置条件（不满足就不要开 v0.25/0.26）**：

- v0.24.3 SOP 全部 exit criteria 通过（legacy `html`/@prop 删除、renderer parity、gate 全绿、docs truth 收敛）。
- Public API surface 已锁定：不再暴露旧模板/装饰器；发布顺序/版本号一致。
- v0.24.3 release notes 与实际 gate 结果一致，确保路线图基于真实状态。

## 一、v0.25.0 — 声明式 DX

### 四个任务，三个是机械改动

> 所有任务默认遵守：不破坏 v0.24.3 新状态；不引入品牌前缀；不新增隐形耦合。

#### TG-01: BuildPipeline 声明式 API (P0)

```
当前:  build.ts → build-client.ts → build-ssg.ts 三阶段硬编码
目标:  lessPipeline({ phases, routes, i18n, output }) 单一配置入口
性质:  纯重构，零 API 破坏
```

不改任何逻辑。把三阶段调用包装在 `BuildPipeline` 类里，用户侧从分散的 `less()` + ctx 传递变成单一配置对象。

接口设计（精简实用版）：

```typescript
// 框架侧
class BuildPipeline {
  constructor(config: {
    phases?: ('ssr-bundle' | 'client-chunks' | 'ssg')[];
    routes: { dir: string };
    i18n?: { locales: string[]; defaultLocale: string };
    output?: { cleanUrls?: boolean };
  });
  async run(ctx: LessBuildContext): Promise<void>;
}

// 用户侧 vite.config.ts
export default defineConfig({
  plugins: [lessPipeline({ routes: { dir: 'app/routes' }, i18n: { locales: ['en', 'zh'] } })],
});
```

现有 `less()` 函数保留兼容，标记 `@deprecated`。

约束：

- 不改变现有 build graph、JSR publish 顺序和输出目录结构。
- CLI / CI 任务名保持兼容，发布工作流无需改动即可通过。

#### TG-02: 路由类型代码生成 (P0)

```
当前:  _getRouteParams() → Record<string, string>，零编译期安全
目标:  构建时扫描 [param] 模式生成 RouteParams 接口
性质:  纯生成，不碰运行时
```

产出：`.less/routes.d.ts`

```typescript
declare module 'virtual:less-routes' {
  interface RouteParams {
    '/blog/[slug]': { slug: string };
    '/registry/[package]/[component]': { package: string; component: string };
  }
}
```

route-scanner 增强：`RouteEntry.params` 字段 + 类型文件生成。不改变路由匹配逻辑。

约束：

- 生成文件路径固定（`.less/routes.d.ts`），不污染源码目录；可被 .gitignore。
- 不引入 runtime 依赖；纯构建期产物。

#### TG-03: `static head` 元数据 (P1)

```
当前:  <open-layout head-extras='<title>...</title><meta...>'> 字符串拼接
目标:  static head = { title, description, ogImage } 自动注入
性质:  简单 parser + SSG 注入点
```

实现：

- 路由组件声明 `static head`
- SSG 构建时读取并注入 `<head>`
- `less-layout` 不再需要 `head-extras` 属性
- 保持 `less-layout` 的 head-extras 兼容（过渡期）

```typescript
class MyPage extends DsdElement {
  static head = {
    title: 'My Page',
    description: 'Page description',
  };
  // SSG build → <title>My Page</title><meta name="description" content="...">
}
```

不需要 OG image / JSON-LD 等复杂场景——以后按需加字段。

约束：

- 保持 `less-layout` 现有 head-extras 兼容，不破坏现有页面。
- SSR/SSG 输出需经过 snapshot 测试（title/meta 落地）。

#### TG-04: `static client` 替代 `island()` (P1)

```
当前:  export default island('my-widget', MyWidget, { strategy: 'visible' })
目标:  class MyWidget extends DsdElement { static client = { strategy: 'visible' } }
性质:  语法糖，不改 island 运行时
```

实现：`static client` 声明 → build-client.ts 扫描时识别 → 自动注册为 island chunk。`island()` 函数保留兼容。

约束：

- 不改变 island chunk 命名/输出结构，避免 breaking change。
- CLI flags、发布工序不受影响。

## 二、v0.26.0 — 响应式务实改进

### 诚实：不做跨组件 SignalContext

理由：

1. `computed()` 当前 0 处生产使用。连基础响应式都没人用，加跨组件订阅是过度设计。
2. LessJS 的组件模型是 **DSD-first 静态组件 + Island 交互**。大多数组件不需要响应式。需要的地方用 component-local `signal()` 已足够。
3. Cross-component state 用 props 传递在 shadow DOM 场景下是最正确的——它尊重组件边界的自然语义。

### 做什么

#### TG-01: Route Params Reactive (P0)

```
当前:  SPA 导航后 _getRouteParams() 仍是旧值
目标:  useRouteParams() 随导航自动更新
```

这是 LessJS 自己网站就需要的功能——SPA 导航浏览 blog、docs 时，URL 变了但页面内容不变。

实现：

```typescript
// 新增
function useRouteParams(): Readonly<Record<string, string>> {
  // 内部用 signal + onNavigate 订阅
}
```

DsdElement 基类暴露 `this.params`（自动订阅路由变化）：

```typescript
class BlogPost extends DsdElement {
  override render() {
    // this.params 自动随 SPA 导航更新
    const slug = this.params.slug;
    const post = posts.find((p) => p.slug === slug);
    return post ? <article>...</article> : <p>Not found</p>;
  }
}
```

实现方式：`onNavigate()` 回调 → set signal → `effect()` re-render（复用现有 VNode tracking）。

#### TG-02: Fine-Grained DOM (`data-keep-alive`) (P1)

```
当前:  signal 变更 → render() → renderToDOM() 全量替换 shadow DOM
问题:  input focus 丢失，CSS transition 打断
目标:  data-keep-alive 属性的元素跳过 DOM 替换，只 patch 属性
```

这是修 bug，不是新功能。任何有 `<input>` 的响应式组件都受影响。

实现：在 `effect()` 的 re-render 路径中，检测 `data-keep-alive`：

```typescript
// renderToDOM 或 _renderIntoShadowRoot 中
if (existingEl.hasAttribute('data-keep-alive') && newEl.tagName === existingEl.tagName) {
  // patch attributes only, preserve DOM node
  for (const [attr, value] of Object.entries(newProps)) {
    if (value == null) existingEl.removeAttribute(attr);
    else existingEl.setAttribute(attr, String(value));
  }
  // skip removeChild + appendChild
  return;
}
```

```
当前:  computed() 有实现但无人用，文档缺失
目标:  补充 computed 使用文档，降低使用门槛
```

不是加功能。是把已有功能让人知道。
