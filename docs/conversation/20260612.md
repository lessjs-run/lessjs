# Nitro Integration Discussion — 2026-06-12

## Context

This document captures the discussion about integrating
[Nitro](https://nitro.build) into openElement, informed by Fresh framework's
public roadmap and our own analysis of multi-platform deployment needs.

## What is Nitro

Nitro is a universal server framework from the Nuxt ecosystem (created by Pooya
Parsa / pi0). It powers Nuxt, SolidStart, and TanStack Start. Key facts:

- 10.6K stars, 805 forks on GitHub
- Generates optimized `.output/` from a single codebase, deployable to **25+
  platforms** with zero config auto-detection:
  Node.js, Bun, Deno, Cloudflare Workers/Pages, Vercel, Netlify, AWS Lambda,
  Deno Deploy, and more
- Runs as a **Vite plugin** — `vite build` produces both frontend + backend
- v3 is in public beta: ground-up rewrite using Rolldown + Vite v8 + web
  standards (srvx replaces h3 as the server layer)
- Built-in caching (ESR/ISR/SWR), file-system API routing, storage drivers,
  WebSocket support (via CrossWS)

## Why this matters

### Fresh is going this way

Luca Casonato (Fresh creator) announced in
[freshframework/fresh#3830](https://github.com/freshframework/fresh/discussions/3830)
that Fresh will adopt Nitro for multi-platform deployment:

> It will also enable Fresh to be deployed to non-Deno systems, like Node or
> Cloudflare Workers, and integrate with various deployment platforms using
> Nitro.

Fresh is currently Deno-only. With Nitro:
- Fresh handles rendering / SSR / island architecture (what it already does)
- Nitro handles the universal server layer and deployment preset logic
- Output deploys to Node, Deno, Cloudflare, Vercel, etc. with minimal config

### openElement has the same constraint

openElement is currently a Deno workspace (20 packages, JSR publishing). Adding
multi-platform deployment would:
- Unlock Node.js, Cloudflare Workers, Vercel, and 22 other deployment targets
- Remove the "Deno only" label
- Match or exceed Fresh's deployment story

Nitro is the right path because:
1. We don't need to build our own server abstraction (srvx already handles this)
2. We don't need to build deployment presets (25+ exist, auto-detected)
3. Nitro v3 uses Rolldown + Vite v8 — same stack as our adapter-vite already
4. It's proven infrastructure (Nuxt, SolidStart, TanStack Start)

## Proposed Architecture

```
openElement JSX / DSD Renderer
        │
        ▼
   Nitro server entry
        │
        ▼
   Nitro preset auto-detection
        │
   ┌────┼────┬──────────┬─────────┐
   ▼    ▼    ▼          ▼         ▼
 Node  Deno Cloudflare  Vercel   AWS Lambda  (25+ platforms)
```

Our job: write a Nitro preset that knows how to:
1. Mount the openElement SSR handler as a Nitro server entry
2. Handle static assets (`www/public/`) via Nitro's `serveStatic`
3. Map DSD / island hydration assets to the correct output paths
4. Expose a `nitro.config.ts` preset so users get zero-config deploy

The rendering layer (JSX → VNode → DSD) stays exactly as-is. We're only
replacing `Deno.serve(app.handler())` with a universal Nitro handler.

## Key Technical Decisions

### Nitro v3 vs v2

Use v3 (current beta, single `./output` for all platforms). v2 is stable but
v3 has the Rolldown / Vite v8 stack that aligns with our adapter-vite direction.

### Server entry approach

Nitro v3 uses srvx as the server runtime. The server entry is a module that
exports a `fetch`-compatible handler. Our integration needs to:

```ts
// server/index.ts (Nitro server entry)
export default defineNitroServerHandler((event) => {
  const app = createApp(); // openElement app instance
  const request = toWebRequest(event);
  const response = await app.handler(request);
  return toNitroResponse(response);
});
```

### Island and DSD hydration under Nitro

- DSD HTML output is server-rendered and works without JS on the client — no
  platform-specific work needed
- Island hydration scripts (`defineIsland()` → Custom Elements) need to be
  bundled correctly per platform (e.g., ESM for Workers, standard for Node)
- Vite handles client-side bundling — Nitro doesn't interfere

### JSR publishing

openElement currently publishes to JSR. Nitro doesn't change this — packages
still ship via JSR. Nitro only affects the **runtime deployment** of the user's
application, not the framework publishing pipeline.

## Follow-up Decision: Protocol-First Runtime Architecture

After reviewing Nitro, Vike, and the current openElement roadmap, we refined the
direction:

- Nitro should not be treated as a small optional adapter only. It is a strong
  candidate for openElement's default production runtime and deployment engine.
- Vite should remain the default module graph, plugin, client build, and HMR
  engine.
- openElement should not spend roadmap budget recreating deployment presets,
  server bundle formats, storage/cache driver plumbing, or platform adapters
  that Nitro already maintains.
- This does not mean openElement becomes a Nitro skin. Nitro owns runtime and
  deployment; openElement owns application, route, render, component, island,
  and UI semantics.

The resulting product model:

```text
openElement = Web Components application framework
default build engine = Vite
default production runtime/deployment engine = Nitro
default component model = Web Components
default render mode = shadow/DSD
first-class opt-in render mode = light DOM
replacement boundary = @openelement/protocols
```

The product should no longer be described only as "DSD-first." DSD/shadow is the
default render mode, but openElement's product identity is broader: a Web
Components application framework with protocol-first internals.

## Vike Comparison

Vike is the closest architectural comparison. It is Vite-native,
renderer-flexible, server-agnostic, and deployment-agnostic. openElement should
borrow that model without cloning Vike's product identity.

The distinction:

```text
Vike = Vite-first, renderer-agnostic meta-framework.
openElement = Web Components-first, Vite+Nitro-powered app framework with
              renderer/runtime protocols.
```

Vike Universal Deploy is valuable as a long-term design signal. Nitro is more
immediately useful because it already supplies production runtime, output, and
platform preset machinery.

## Replacement Boundaries

The protocol layer should make these modules replaceable over time:

- renderer: default openElement JSX/VNode/render-node; future React/Vue/Svelte/Lit
  renderer adapters possible;
- component adapter: default DsdElement/custom elements; future Lit, plain custom
  elements, framework island adapters possible;
- route manifest: default file routes; future manual route manifest or CMS route
  source possible;
- islands: default openElement island metadata and Vite client entries; future
  framework-specific island adapters possible;
- runtime: default Nitro; future Hono-only, Cloudflare-native, Node-native, or
  Vike-style deployment metadata possible;
- cache/storage: default Nitro primitives; future KV, R2, Redis, filesystem,
  memory, or database-backed adapters possible;
- deployment: default Nitro presets; future provider-specific or Universal Deploy
  style adapters possible.

Conformance tests are required. A protocol without a baseline implementation
and test suite is not considered real.

## Roadmap Change

The route changed from "v0.37.5 Protocol Ports + DaisyUI Coverage Completion" to:

1. **v0.37.4** remains publish recovery. JSR visibility and consumer smoke are
   still release truth.
2. **v0.37.5** becomes Protocol-First Runtime Architecture. It accepts ADR-0096,
   makes `@openelement/protocols` the replacement boundary, and prepares the
   Nitro proof plan.
3. **v0.37.6** becomes Vite + Nitro Runtime Proof. It proves Node and Cloudflare
   Workers output without replacing openElement route/render/component semantics.
4. **v0.38.x** becomes product/package surface reset based on protocol and Nitro
   evidence.

The daisyUI interactive/form component coverage remains valid backlog, but it
is moved behind the runtime/protocol pivot.

## JSR Release Risk

The JSR publish incident should be treated as external registry instability, not
as ordinary openElement CI slowness. Direct registry checks remain useful for
release notes and distribution telemetry, but they no longer decide whether a
version can exit. Future release design should also consider fallback publishing
or distribution options so JSR is not a single point of release truth.

## Follow-up Release Policy: JSR Best-Effort

The release policy was refined again after the JSR incident: live JSR
publication is no longer a version-exit criterion. openElement should continue
to run local/CI publish attempts and record the registry state, but roadmap
execution should not stop on JSR visibility, metadata propagation, or
post-publish JSR consumer smoke.

Version exit is now based on repository-controlled evidence:

- SOP tasks and acceptance criteria;
- local gates, including package graph and publish dry-run;
- AutoFlow evidence where required;
- dev/main non-JSR CI;
- changelog, release notes, docs, package versions, and governance alignment.

Release notes must be honest about JSR availability. They can state that JSR
distribution is delayed or partially visible, but they must not claim full JSR
availability without direct resolver evidence.
