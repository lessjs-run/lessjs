# SOP: v0.17.4 - Compatibility Boundary Hardening

> Purpose: make the v0.17.3 compatibility boundary enforceable in code before
> the project moves into the v0.18 CEM and install flow.
>
> Release type: **patch (correctness and build robustness)**.

## Entry Criteria

- v0.17.3 docs, ADR, roadmap, and changelog are complete.
- The project still treats third-party UI libraries as client-only unless
  explicitly admitted for SSR.
- Current Shoelace/Media Chrome style failures are understood as SSR admission
  problems, not proof that every library needs DOM simulation.

## Problem To Fix

Browser-dependent islands or package components can enter the SSR bundle before
their `ssr: false` or compatibility metadata is applied. Once imported, they
may register real custom element classes and nested rendering can try to SSR
browser-only tags.

The fix must happen before SSR entry generation, not after registration.

## Step-by-Step Execution

### Step 1: Audit Current Discovery Paths

Inspect these files first:

- `packages/adapter-vite/src/entry-descriptor.ts`
- `packages/adapter-vite/src/entry-renderer.ts`
- `packages/adapter-vite/src/route-scanner.ts`
- `packages/core/src/render-dsd.ts`
- `packages/core/src/types.ts`

Record where each island source becomes an SSR import:

- local island file
- package manifest island
- nested custom element found in rendered HTML

### Step 2: Add An SSR Admission Plan

Create a structured plan before generating SSR entry code.

The plan must include:

```ts
interface SsrAdmissionPlan {
  renderableTags: string[];
  clientOnlyTags: string[];
  rejectedTags: string[];
  reasons: Record<string, string>;
}
```

Rules:

- `less.ssr === false` -> `clientOnlyTags`
- package manifest without known SSR capability -> `clientOnlyTags`
- duplicate tag -> `rejectedTags`
- known renderer and `ssr: true` -> `renderableTags`

### Step 3: Apply The Plan Before Imports

Generated SSR entry code must only import modules for `renderableTags`.

Do not rely on:

- redefining `customElements`
- replacing classes after module import
- catching render errors as the main compatibility mechanism

### Step 4: Guard Nested Rendering

Update nested custom-element rendering so `clientOnlyTags` are skipped even
when they appear inside the HTML produced by a parent component.

Expected behavior:

- parent can SSR
- child client-only tag remains in light DOM or placeholder output
- no attempt is made to instantiate the client-only child class on the server

### Step 5: Extend DSD Report

`dsd-report.json` must record:

- skipped tag name
- skip reason
- source: local island, package island, nested tag
- whether hydration/client registration metadata was emitted

### Step 6: Add Fixtures

Add focused fixtures instead of relying on the public docs site:

- local island with `less.ssr = false`
- package island with `ssr: false`
- parent component that outputs a client-only child tag
- browser-dependent mock component that throws if imported in SSR

- [x] nested client-only child tag is skipped without stack traces

## Completion Notes

- `packages/adapter-vite/src/entry-descriptor.ts` owns the
  `SsrAdmissionPlan`.
- `packages/adapter-vite/src/entry-renderer.ts` imports only SSR-admitted
  islands and exposes client-only tags to the runtime.
- `packages/adapter-vite/src/route-scanner.ts` reads local `less` metadata
  statically and skips browser-only package manifest imports.
- `packages/core/src/render-nested.ts` skips client-only nested tags.
- `packages/adapter-vite/src/cli/build.ts` exits cleanly after successful
  one-command builds.
- Remaining browser-only package support is intentionally deferred to v0.18
  manifest/CEM validation.

## Exit Criteria

- SSR entry generation is driven by `SsrAdmissionPlan`.
- Client-only tags never become server imports by accident.
- Build reports explain skipped tags.
- v0.18 can start without carrying this compatibility bug forward.

## Non-Goals

- Do not implement CEM parsing here.
- Do not implement `less add`.
- Do not implement DOM simulation.
- Do not promise Shoelace/Media Chrome SSR support.
