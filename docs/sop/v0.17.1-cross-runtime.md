## Changes Made

### ssg.ts тАФ `Deno.build.os` тЖТ `process.platform`

Before:

```ts
const bundleUrl = Deno.build.os === 'windows'
  ? 'file:///' + bundlePath.replace(/\\/g, '/')
  : 'file://' + bundlePath;
```

After:

```ts
const bundleUrl = process.platform === 'win32'
  ? 'file:///' + bundlePath.replace(/\\/g, '/')
  : 'file://' + bundlePath;
```

Rationale: `process.platform` is available in Deno (via `node:process` compat),
Node.js natively, and Bun natively. `'win32'` is the Node.js convention
for Windows тАФ equivalent to `Deno.build.os === 'windows'`.

### ssg.ts тАФ `Deno.exit(1)` тЖТ `process.exit(1)`

Before:

```ts
Deno.exit(1);
```

After:

```ts
process.exit(1);
```

Rationale: `process.exit()` works identically across Deno, Node, and Bun.
The file already imports `process from 'node:process'`.

## Post-v0.17.1 Outlook

With this patch, adapter-vite's SSG CLI has **zero hard Deno dependencies**.
The only remaining `Deno.*` reference is the guarded `Deno.readTextFileSync`
in `workspace-alias.ts` which falls back to `node:fs`.

This means:

- **npm/bun users can run** `npx lessjs build` or `bunx lessjs build`
  (JSR auto-syncs to npm, core packages are runtime-agnostic).
- **SSG CLI** can run under `bun run ssg.ts` or `node ssg.ts`
  (with `--import-map` for Deno-specific import resolution).
- Future work: getting-started docs for npm/bun users, CI matrix testing
  across runtimes.
