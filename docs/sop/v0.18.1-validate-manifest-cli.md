# SOP: v0.18.1 - `less validate-manifest`

> Purpose: provide a deterministic validation command for package manifests and
> CEM-derived metadata before any package is installed, registered, or rendered.
>
> Release type: **minor patch (CLI + diagnostics)**.

## Entry Criteria

- v0.18.0 CEM parser exists.
- Compatibility tiers exist: `ssr-capable`, `client-only`, `rejected`,
  `experimental-dom`.
- Validation can run without executing third-party component modules.

## Step-by-Step Execution

### Step 1: Define Validation Output

Add a stable JSON output shape:

```ts
interface ManifestValidationReport {
  packageName?: string;
  version?: string;
  valid: boolean;
  compatibility: 'ssr-capable' | 'client-only' | 'rejected' | 'experimental-dom';
  errors: ValidationDiagnostic[];
  warnings: ValidationDiagnostic[];
  tags: ValidatedTag[];
}
```

Each diagnostic must include:

- code
- severity
- message
- tag name when applicable
- file/module path when applicable
- actionable fix

### Step 2: Validate Schema And Tags

Check:

- CEM schema shape
- Less extension shape
- custom element tag naming rules
- duplicate tag names
- required module paths
- unsupported or unknown SSR declarations

### Step 3: Validate Module Paths Without Execution

Resolve paths from package root, but do not import third-party modules.

Reject:

- absolute paths outside package root
- path traversal
- missing modules
- ambiguous exports
- duplicate exports for the same tag

### Step 4: Classify Compatibility

Apply the v0.18.0 tier rules:

- explicit supported SSR renderer -> `ssr-capable`
- CEM-only package -> `client-only`
- invalid package -> `rejected`
- explicit opt-in DOM simulation candidate -> `experimental-dom`

### Step 5: Add CLI

Command:

```sh
less validate-manifest ./custom-elements.json
less validate-manifest npm:@scope/package --json
less validate-manifest ./custom-elements.json --strict
```

Required behavior:

- human-readable default output
- `--json` machine-readable output
- non-zero exit code when `valid === false`
- strict mode fails warnings that would block Hub admission

## Verification

- [ ] valid Less manifest returns exit code 0
- [ ] CEM-only package is valid but `client-only`
- [ ] duplicate tag returns exit code 1
- [ ] path traversal returns exit code 1
- [ ] malformed Less extension returns actionable error
- [ ] JSON output is deterministic
- [ ] `deno task fmt && deno task lint && deno task typecheck`
- [ ] `deno task test`
- [ ] `deno task build`

## Exit Criteria

- CI can use `less validate-manifest --json`.
- The Hub can ingest the same report shape.
- `less add` can call validation before changing project files.

## Non-Goals

- Do not install packages.
- Do not execute package modules.
- Do not render components.
