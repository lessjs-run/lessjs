# SOP: v0.17.3 - Multi-Framework Adapters + Compatibility Boundary

> Purpose: add framework-specific SSR adapters for vanilla Web Components and
> React-to-Web-Component bridging, while defining the third-party UI library
> compatibility boundary.
>
> Release type: **minor (new packages, no breaking change)**.

## Goal

1. **adapter-vanilla**: SSR adapter for plain/vanilla Web Components whose
   `render()` returns a string directly. Provides DSD hydration base class and
   auto-registration.
2. **adapter-react**: SSR adapter for React components wrapped as Web
   Components. Handles `React.createElement` -> string conversion for DSD
   rendering without a DOM.
3. **compatibility boundary**: do not treat every registered custom element as
   SSR-safe. Browser-dependent packages must use client-only fallback unless a
   validated manifest or adapter proves otherwise.

## Entry Criteria

- v0.17.2 SSR filtering and `dsd-report.json` manifest decisions exist.
- `RendererProtocol` can register more than one adapter family.
- The release must stay additive: no breaking changes to existing packages.

## Implementation Details

### Scope 1: adapter-vanilla

Plain Web Components that `render(): string` already work with core's
`renderDSD()`. The adapter adds:

- `installVanillaAdapter()`: registers the `vanilla` adapter via
  `registerAdapter()`.
- `extractVanillaStyles()`: extracts static `styles` from component classes.
- `DsdVanillaElement`: base class for DSD detection, hydration event binding,
  and cleanup.
- zero framework dependencies.

### Scope 2: adapter-react

React-to-Web-Component wrappers render through React's server renderer:

- `installReactAdapter()`: registers the `react` adapter.
- `isReactElement()`: detects React elements.
- `renderReactToString()`: uses `ReactDOMServer.renderToStaticMarkup()`.
- `DsdReactElement`: wrapper base class for SSR string output and client-side
  mounting.
- `react` and `react-dom` remain peer dependencies.

### Scope 3: Compatibility Boundary

v0.17.3 is not a claim that Shoelace, Media Chrome, or any arbitrary Web
Component package can be SSR-rendered automatically.

The supported statement is narrower:

- Lit, vanilla string renderers, and React wrappers can be SSR-rendered through
  declared adapters.
- Third-party UI libraries that depend on real browser DOM APIs are
  client-only unless their package manifest explicitly passes a LessJS SSR
  admission gate.
- A local island with `less.ssr === false` must not be imported or registered
  inside the SSR bundle.
- Nested custom-element rendering must respect client-only decisions, otherwise
  a parent wrapper can accidentally trigger SSR for browser-only child tags.

This policy is captured in
`docs/adr/0028-universal-ssr-via-dom-simulation.md` and drives v0.18 planning.

## Non-Goals

- Do not promise SSR for every CEM package.
- Do not use package registration alone as proof of SSR safety.
- Do not silently fall back to DOM simulation.
- Do not call third-party UI library demos "SSR-compatible" until build reports
  prove they render without SSR errors.

## v0.17.4 Follow-up Gate

Before v0.18 starts, a compatibility hardening patch should close the gap
exposed by browser-dependent UI libraries:

- read local island `less` metadata before SSR entry generation
- exclude `ssr: false` local islands from SSR imports
- block package SSR registration unless the manifest declares a supported
  renderer/capability
- make nested custom-element rendering respect client-only tags
- make `dsd-report.json` include skipped tags and skip reasons
- add fixtures for browser-dependent tags such as Shoelace-style components
