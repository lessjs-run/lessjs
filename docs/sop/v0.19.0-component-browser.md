# SOP: v0.19.0 (Phase 2) — Component Browser + Full-Stack Usage Workflow

> Purpose: transform the Registry Hub from a "package list" into a complete
> discovery → evaluation → installation → usage pipeline, with component-level
> drill-down, rendered previews, and a working `less add` CLI.
>
> Release type: **minor patch (new routes + CLI integration)**.
>
> **This is the most important deliverable for the project** — it closes the
> final gap between what the Hub shows and what developers can actually do.

## Strategy

v0.19.0 proved the Hub architecture works (static SSG, PR-based submission, CI
validation). v0.19.1 makes the Hub **useful**:

| Gap                                      | v0.19.1 Solution                                |
| ---------------------------------------- | ----------------------------------------------- |
| `less add` CLI doesn't exist             | Implement `less add` from v0.18.2 SOP spec      |
| Component tags are plain text            | Clickable → opens component detail page         |
| No rendered preview of components        | SSR snapshots + inline HTML preview             |
| No usage examples                        | Per-component code snippets                     |
| No "what do I do after install" guidance | Usage section on every detail page              |
| Only one user persona considered         | 4 personas: builder, author, evaluator, browser |

## User Personas

### Persona A: Alex — LessJS Site Builder

- **Goal**: "I need `<sl-dialog>` in my LessJS page."
- **Journey**: Browse → Find package → Click component → See preview + snippet
  → `less add` → Copy snippet → Done
- **v0.19.1 must**: make `less add` work, show rendered preview, show snippet

### Persona B: Bob — WC Package Author

- **Goal**: "I want LessJS users to discover my components."
- **Journey**: `less hub submit` → CI validates → PR merges → Users browse
  → Users click components → See beautiful rendered previews
- **v0.19.1 must**: ensure submitted packages get good component previews

### Persona C: Carol — Framework Evaluator

- **Goal**: "Does LessJS have a healthy component ecosystem?"
- **Journey**: Browse Hub → See rich package list → Click a package → See
  per-component previews → Confident to commit
- **v0.19.1 must**: present a polished, content-rich Hub that inspires trust

### Persona D: Dave — Casual Browser

- **Goal**: "What kinds of Web Components exist?"
- **Journey**: Browse → Click interesting package → Click component → See live
  preview → Gets inspired
- **v0.19.1 must**: make browsing visually rewarding

## Step-by-Step Execution

### Step 1: Implement `less add` CLI (P0)

**Files to create/modify**:

- `packages/hub/src/cli/less-add.ts` — new file, main CLI command
- `packages/hub/mod.ts` — export new CLI entry
- `packages/hub/__tests__/less-add.test.ts` — tests

**Behavior**:

```
less add @shoelace-style/shoelace
```

1. Resolve package source: check JSR first, then npm, then local path
2. Run validation (reuse v0.18.1 `validate-manifest` logic via hub's builder)
3. Determine compatibility tier (ssr-capable / client-only / rejected)
4. Print summary to stdout:
   ```
   Package: @shoelace-style/shoelace@2.20.1
   Source: npm
   Compatibility: client-only
   Components: 39 (0 SSR-capable, 39 client-only)
   Warnings: 1 — No LessJS SSR metadata found
   Config changes:
     packages["@shoelace-style/shoelace"] — client-only (no SSR)

   To use <sl-button> in a route:
     import '@shoelace-style/shoelace';
     // Use in template:
     // <sl-button variant="primary">Click me</sl-button>
   ```
5. On `--apply` flag: actually update project config (dry-run by default)

**Acceptance**: `deno run -A packages/hub/src/cli/less-add.ts @shoelace-style/shoelace`
prints valid output without crashing.

### Step 2: Create Component Detail Route (P0)

**New file**: `www/app/routes/registry/[package]/[component].ts`

Route pattern: `/registry/:package/:component`

```
www/app/routes/registry/
├── _hub-data.ts                  ← Generated index (exists, renamed)
├── _hub-data-full.ts             ← Generated full records (exists, renamed)
├── _renderer.ts                  ← Layout renderer (exists, enhanced)
├── index.ts                      ← Package list (exists, enhance)
├── [package].ts                  ← Package detail (exists, enhance)
└── [package]/
    └── [component].ts            ← Component detail (NEW)
```

**Acceptance**: `/registry/@shoelace-style~shoelace/sl-dialog` renders a valid
SSG page at build time.

**Implementation**:

1. Create `www/app/routes/registry/[package]/` directory
2. Create `[component].ts` in it
3. `getStaticPaths()` assembles all (package, component) pairs from hub data
4. Route component takes `package` + `component` params
5. Looks up component data from hub-data-full.ts
6. Renders: breadcrumb, component tag name, compatibility badge, rendered
   preview, usage snippet, API reference (from CEM), back link to package

### Step 3: Add Component Drill-Down Links to Package Detail (P0)

Modify `[package].ts`:

- Each component tag in the "Components" section becomes a clickable link:
  `<a href="/registry/${packagePath}/${tag.tagName}">${tag.tagName}</a>`
- Add hover effect and visual indicator that items are clickable

### Step 4: Generate SSR Snapshots for Component Previews (P1)

Modify `packages/hub/src/snapshot.ts`:

- Add function to render individual component in isolation
- Output is a clean HTML fragment (<5KB) stored in the HubPackageRecord
- For SSR-capable components: full rendered HTML with DSD template
- For client-only components: placeholder with tag name and note
- Wire into `hub:scan` pipeline to populate `ssrSnapshot` field per tag

Store snapshots in the package record's tag array:

```typescript
interface HubTagRecord {
  tagName: string;
  compatibility: string;
  validationErrors: number;
  validationWarnings: number;
  ssrSnapshot?: string; // HTML fragment for inline preview
  // NEW:
  attributes?: CemAttribute[]; // From CEM manifest
  events?: CemEvent[];
  slots?: CemSlot[];
  usageExample?: string; // Auto-generated or hardcoded
}
```

### Step 5: Add Usage Examples to Detail Pages (P1)

Enhance both `[package].ts` and `[package]/[component].ts`:

**Package-level usage section**:

- Shows `less add <package>` command with copy button (text-selectable)
- Shows import statement
- Links to component-level pages for per-component details

**Component-level usage section**:

- Shows isolated usage example:
  ```html
  <sl-dialog label="Hello" open>
    This is a dialog.
    <sl-button slot="footer" variant="primary">Close</sl-button>
  </sl-dialog>
  ```
- SSR considerations: "This component is client-only. Use `<less-client-only>`
  wrapper if rendering server-side."

### Step 6: Enhance Package List with Badges and Metrics (P2)

Modify `index.ts`:

- Show total component count per package on the card
- Show SSR-capable vs client-only breakdown
- Add sort options: by name, by component count, by last updated
- Add "New" badge for recently submitted packages (<7 days)

### Step 7: API Reference from CEM (P2)

For components with CEM data:

- Display attributes table (name, type, default, description)
- Display events table (name, detail type, description)
- Display slots (name, description)
- Store CEM data in HubPackageRecord during `hub:scan`

Generic fallback when no CEM:

- Show only tag name, compatibility, and validation results
