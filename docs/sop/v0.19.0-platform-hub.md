# SOP: v0.19.0 - Registry Hub MVP

> Purpose: build the first public WC Registry Hub on top of validation and
> build artifacts, not on blind package execution.
>
> Release type: **minor (new subsystem, backward compatible)**.

## Goal

1. **Validated package index**: searchable records sourced from manifests,
   compatibility reports, and CI artifacts.
2. **Package detail pages**: show tags, module paths, compatibility tier,
   bundle size, hydration behavior, and known risks.
3. **Snapshot previews**: only for packages/components that pass the v0.18
   admission gate.
4. **Install guidance**: generate `less add` commands and warnings from the
   same validation data used by the engine.

## Entry Criteria

- v0.18.1 validation reports are stable enough for machine ingestion.
- v0.18.2 `less add` dry-run output exists.
- Hub MVP can be built without executing arbitrary package code at request time.

## MVP Scope

| # | Item                          | Location                               | Estimate   |
| - | ----------------------------- | -------------------------------------- | ---------- |
| 1 | Hub data schema               | `packages/hub/schema/`                 | 1 session  |
| 2 | Static/API package index      | `packages/hub/`                        | 2 sessions |
| 3 | Search + package detail UI    | `packages/hub/web/` or `www/registry/` | 2 sessions |
| 4 | Report ingestion              | `packages/hub/ingest/`                 | 2 sessions |
| 5 | Snapshot artifact display     | `packages/hub/snapshots/`              | 2 sessions |
| 6 | Security and conflict summary | `packages/hub/audit/`                  | 1 session  |

## Hub Record

Each Hub package version should be derived from machine-readable artifacts:

```ts
interface HubPackageVersion {
  name: string;
  version: string;
  source: 'jsr' | 'npm' | 'local';
  manifestHash: string;
  tags: HubTagRecord[];
  compatibility: 'ssr-capable' | 'client-only' | 'rejected' | 'experimental-dom';
  reports: {
    validation: string;
    dsd?: string;
    bundle?: string;
    security?: string;
  };
}
```

The Hub can display README content, but README content is never a trust
decision.

## Step-by-Step Execution

1. Define Hub record schema from validation/build artifacts.
2. Add ingestion for local fixture reports first.
3. Add search index over package name, tags, and compatibility tier.
4. Add package detail page or API response.
5. Add install guidance using `less add --dry-run` equivalent data.
6. Add rejected/client-only display states before preview features.

## Explicit Deferrals

These are valuable but not part of the first Hub MVP:

- publisher accounts
- package signing
- paid marketplace features
- scoped Custom Element Registries
- edge rendering adapters
- plugin marketplace
- automatic package execution in the Hub service

## Verification

- [ ] `less validate-manifest` output can be ingested as a Hub record
- [ ] `dsd-report.json` can be linked to a package version
- [ ] search returns packages with compatibility tier and install warning
- [ ] package detail pages distinguish SSR-capable and client-only components
- [ ] rejected packages show actionable reasons
- [ ] snapshot previews are absent for components that did not pass admission
- [ ] `deno task fmt && deno task lint && deno task typecheck`
- [ ] `deno task test`
- [ ] `deno task build`

## Exit Criteria

- Hub can ingest validation and build reports.
- Hub can show package compatibility without trusting README text.
- Hub can show install guidance and warnings.
- v0.19.1 can add previews and quality gates on top of stable Hub records.

## Key Risk

The Hub creates governance and supply-chain responsibilities. The MVP should be
an evidence viewer first. Marketplace, signing, and publisher workflows should
wait until validation quality and abuse policy are proven.
