# SOP: v0.19.0 - Registry Hub + Submission Pipeline

> Purpose: ship the first public-facing Registry Hub with package search,
> compatibility evidence, and a CLI-driven submission pipeline.
>
> Release type: **minor (new subsystem, backward compatible)**.
>
> **One-shot release** — Hub MVP + `less hub submit` CLI + GitHub PR
> automation, no further v0.19.x splitting.

## Strategy

v0.19.0 delivers the entire Hub subsystem in one release:

| Capability                     | Why it belongs together                                                 |
| ------------------------------ | ----------------------------------------------------------------------- |
| Hub data schema + builder      | Foundation — everything else depends on stable Hub record types         |
| Search index + registry UI     | User-facing value — searchable package list with compatibility evidence |
| Package detail pages           | Deep inspection of compatibility, install warnings, snapshots           |
| `less hub submit` CLI          | Turnkey submission: validate → bundle → PR                              |
| GitHub Actions auto-validation | CI gate for incoming submissions                                        |
| Snapshot evidence display      | SSR/SSG preview thumbnails for admitted packages                        |
| Fixture data for demo          | Shoelace, Media Chrome, React showcase as living Hub examples           |

**Why not split?** Each piece alone is useless without the next. A search UI
without submission has nothing to search. A CLI without CI has no trust. A
CI without UI has no discoverability.

## Entry Criteria

- v0.18.3 release complete (all packages tagged v0.18.3)
- `less validate-manifest` CLI produces stable JSON output
- `less add` dry-run output exists
- dsd-report.json schema is stable
- `deno task build` produces deployable artifacts
- GitHub CLI (`gh`) is available for PR automation

Original blockers (pre-audit):

- `less hub submit` must have a reachable PR mode, not only default dry-run.
- The CLI must call the real manifest validation pipeline and preserve its
  diagnostics.
- Hub records must compute and validate `manifestHash`.
- Hub CI must use real Deno validator tasks, reject malformed records, reject
  missing artifacts, check duplicate tags against the actual index shape, and
  regenerate/diff-check `hub-index/index.json`.
- Registry detail pages must be statically emitted for scoped and unscoped
  package names.
- The stale `<less-registry-search>` renderer injection must either be removed

- Root `typecheck`, CI, publish, and publish dry-run flows must include
  `@lessjs/hub`.

## Data Model

### Hub Package Record (hub-index/packages/*.json)

```ts
/** Per-version record stored in hub-index/packages/<scope>/<name>.json */
interface HubPackageRecord {
  /** Package identity */
  schema: 'hub-package-v1';
  name: string; // e.g. "shoelace"
  scope: string; // e.g. "@shoelace" or ""
  version: string; // semver
  source: 'jsr' | 'npm' | 'local';
  repository?: string; // GitHub URL
  description?: string;
  homepage?: string;

  /** Manifest integrity */
  manifestHash: string; // sha256 of custom-elements.json
  manifestPath?: string; // relative path in submission

  /** Compatibility */
  compatibility: CompatibilityTier; // 'ssr-capable' | 'client-only' | 'rejected' | 'experimental-dom'
  compatibilityJustification: string;

  /** Tags discovered from CEM */
  tags: HubTagRecord[];

  /** Evidence reports (relative paths or inline) */
  reports: {
    validation: string | ManifestValidationReport; // from validate-manifest
    dsd?: string | DsdBuildReport; // from dsd-report.json
    bundle?: string | BundleReport; // bundle size info
    security?: string | SecurityReport; // audit info
  };

  /** SSR snapshot paths */
  snapshotPaths: Record<string, string>; // tagName → snapshot path

  /** Install guidance */
  installGuidance: HubInstallGuidance;

  /** Submission metadata */
  submittedAt: string; // ISO 8601
  submittedBy?: string; // GitHub username
  validatorVersion: string; // e.g. "0.19.0"
}

interface HubTagRecord {
  tagName: string; // e.g. "sl-button"
  compatibility: CompatibilityTier;
  validationErrors: number;
  validationWarnings: number;
  ssrSnapshot?: string; // path to snapshot
}

interface HubInstallGuidance {
  safeToInstall: boolean;
  command: string; // e.g. "less add shoelace"
  configChanges: string[]; // config keys that will be added
  warnings: string[]; // known risks
  ssrCapable: boolean;
}

type CompatibilityTier =
  | 'ssr-capable'
  | 'client-only'
  | 'rejected'
  | 'experimental-dom';
```

### Hub Index (hub-index/index.json)

```ts
/** Lightweight search index, regenerated on each merge */
interface HubIndex {
  schema: 'hub-index-v1';
  updatedAt: string;
  packages: HubIndexEntry[];
}

interface HubIndexEntry {
  name: string;
  scope: string;
  version: string;
  description: string;
  compatibility: CompatibilityTier;
  tags: string[]; // tag names only
  source: 'jsr' | 'npm' | 'local';
  safeToInstall: boolean;
  ssrCapable: boolean;
}
```

### Submission Bundle (hub-submission.json)

```ts
/** Produced by `less hub submit`, consumed by Hub CI */
interface HubSubmission {
  schema: 'hub-submission-v1';
  package: HubPackageRecord;
  artifacts: HubArtifact[];
}

interface HubArtifact {
  path: string; // relative path inside bundle
  contentType: string; // "application/json" | "text/html"
  content: string; // base64-encoded or inline
}
```

## CLI Design

### `less hub submit [options]`

```
Usage: less hub submit [options]

Submit a Web Component package to the LessJS Registry Hub.

Options:
  --dir <path>          Package directory (default: cwd)
  --source <type>       Package source: jsr | npm | local (default: local)
  --dry-run             Only validate, do not create PR (default: true)
  --no-pr               Skip GitHub PR, output submission bundle only
  --out <path>          Output path for submission bundle (default: ./hub-submission.json)
  --json                Output as JSON (default: human-readable)
  --verbose             Verbose output
  -h, --help            Show help

Steps:
  1. Read package config (package.json / deno.json / CEM)
  2. Run validate-manifest on CEM
  3. Run build if SSR-capable or experimental-dom, collect dsd-report
  4. Build HubPackageRecord
  5. Bundle artifacts → hub-submission.json
  6. If --dry-run: print submission preview and exit
  7. If not --dry-run: fork repo → commit → PR via gh CLI

Exit codes:
  0: Submission ready
  1: Validation failed
  2: Build failed
  3: Submission bundle invalid
```

## Hub CI Design

```yaml
# .github/workflows/hub-ci.yml
name: Hub Submission CI
on:
  pull_request:
    paths: ['hub-index/**']

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: denoland/setup-deno@v2
      - name: Validate submission schema
        run: deno task hub:validate-submission
      - name: Check duplicate tags
        run: deno task hub:check-duplicates
      - name: Verify artifact integrity
        run: deno task hub:verify-artifacts
      - name: Auto-merge (if safe)
        if: success() && github.actor != 'dependabot[bot]'
        run: |
          PACKAGE=$(gh pr view ${{ github.event.pull_request.number }} --json files --jq '.files[0].path' | grep -oP 'hub-index/packages/\K[^.]+')
          TIER=$(jq -r '.compatibility' "hub-index/packages/$PACKAGE.json")
          if [ "$TIER" = "ssr-capable" ] || [ "$TIER" = "client-only" ]; then
            gh pr merge --auto --squash "${{ github.event.pull_request.number }}"
          fi
```

## Explicit Deferrals

These are valuable but not part of v0.19.0:

- Publisher accounts / auth
- Package signing / code signing
- Paid marketplace features
- Scoped Custom Element Registries
- Edge rendering adapters
- Plugin marketplace
- Automatic package execution in the Hub service
- Version diff / regression comparison UI
- Usage statistics / download counts
- API token management

## Key Risks

| Risk                                     | Mitigation                                                               |
| ---------------------------------------- | ------------------------------------------------------------------------ |
| GitHub API rate limits for PR automation | Implement exponential backoff; document token setup                      |
| Large submission bundles bloat repo      | Keep snapshots small; add size limits in CI                              |
| Spam/fake submissions                    | Auto-merge only for validated, non-rejected packages; manual review gate |
| `gh` CLI not available                   | `--dry-run` is default; `--no-pr` always available                       |
| Hub data becomes stale                   | Regenerate index on each merge; version tracking                         |
| www build time with many packages        | Lightweight index.json, lazy-load detail pages                           |

## Exit Criteria

- `packages/hub/` package exports schema, builder, indexer, submitter
- `less hub submit` CLI works end-to-end (dry-run and PR modes)
- Hub CI validates and conditionally auto-merges submissions
- Registry UI (search + detail) renders from fixture data
- Package detail shows accurate compatibility, install guidance, and snapshot
- All 4 compatibility tiers are visually distinct
- No regressions in existing v0.18.x functionality
- v0.19.1 can add features on top of stable Hub records
