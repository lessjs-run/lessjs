# SOP: v0.18.0 - Universal WC Engine Admission Layer

> Purpose: make LessJS accept third-party Web Component packages through a
> validated compatibility model instead of guessing that every CEM package can
> SSR.
>
> Release type: **minor (new capability, backward compatible)**.

## Goal

1. **CEM parser**: read standard `custom-elements.json` from third-party WC
   packages.
2. **Compatibility tiers**: classify each package/component as SSR-capable,
   client-only, rejected, or experimental.
3. **Conservative default**: CEM without a LessJS SSR extension defaults to
   client-only, not SSR.
4. **Structured diagnostics**: every admission decision appears in validation
   output and `dsd-report.json`.

## Entry Criteria

- v0.17.4 SSR admission planner is complete.
- Client-only tags are excluded from SSR imports before SSR bundle generation.
- Browser-dependent fixtures no longer produce repeated SSR stack traces.
- The roadmap and ADR-0028 still use conservative third-party admission
  language.

## Compatibility Tiers

| Tier               | Meaning                                                            | Build behavior                                                       |
| ------------------ | ------------------------------------------------------------------ | -------------------------------------------------------------------- |
| `ssr-capable`      | Explicit Less manifest or adapter says SSR is supported            | Import/register in SSR bundle and render                             |
| `client-only`      | Browser-only package or no SSR declaration                         | Exclude from SSR bundle, emit client registration/hydration metadata |
| `rejected`         | Invalid manifest, duplicate tag, unsafe path, or unresolved module | Fail before code generation                                          |
| `experimental-dom` | Opt-in DOM simulation candidate                                    | Render only when explicit flag is enabled and report all results     |

## CEM Parser Rules

New module: `packages/core/src/cem-parser.ts`.

Required behavior:

- parse `custom-elements.json` without executing package code
- extract tag names, module paths, exports, attributes, events, slots, CSS
  parts, and CSS custom properties when present
- preserve unknown CEM fields for future compatibility
- reject invalid tag names and unresolved module paths
- detect duplicate tags before SSR registration

Important default:

```ts
// CEM describes a component. It does not prove SSR safety.
less.ssr ??= false;
less.dsd ??= false;
hydrate.strategy ??= 'client';
```

SSR becomes true only when a Less extension, package manifest, or explicit user
configuration declares a supported renderer/capability.

## Step-by-Step Execution

1. Add CEM parser types and fixtures without importing package code.
2. Convert CEM declarations into internal manifest candidates.
3. Add compatibility classifier and tier reasons.
4. Extend SSR/client-only planner to consume CEM-derived candidates.
5. Extend `dsd-report.json` with compatibility tier and skip reason.
6. Add fixtures for SSR-capable, client-only, rejected, and experimental tags.
7. Update docs and changelog only after tests prove the tier behavior.

## Planner Rules

The build planner must run before SSR entry generation.

It decides:

- which modules are safe to import in the SSR bundle
- which tags are rendered by `renderDsd()`
- which tags are skipped as client-only
- which tags need generated client registration
- which manifest entries fail the build

It must not rely on redefining `customElements` after a browser-dependent
module has already been imported.

- [ ] explicit Less SSR metadata enables SSR only when a renderer supports it

## Exit Criteria

- CEM parsing works for fixtures without package execution.
- Unknown CEM packages default to client-only.
- SSR-capable packages require explicit LessJS capability metadata.
- Reports explain every compatibility decision.
- v0.18.1 can build `less validate-manifest` on top of the same classifier.

## Key Risk

The Web Components ecosystem is uneven. Some packages are string-renderable,
some are Lit-renderable, some are browser-only, and many CEM manifests do not
describe runtime constraints. v0.18 succeeds only if the engine is strict about
admission and graceful about fallback.
