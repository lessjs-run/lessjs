# LessJS v0.19.0 Release Closure Step-by-Step SOP

## 0. SOP 目标

将 LessJS v0.19.0 从"有方向、有实现但不 release-ready"的状态，推进到"可验证、可解释、可发布的 release candidate"。

**范围限定**：本 SOP 只覆盖 v0.19.0 release closure。不涉及 v0.20/v0.21 路线图，不涉及新功能开发，不涉及架构重构。所有步骤均为修复、验证、收敛和文档对齐。

**核心原则**：

1. 不允许在 dirty state 下宣布 release-ready
2. 每一步都有明确的验收命令和通过标准
3. 不用扩大 timeout 掩盖问题，不用 Infinity 阈值放行错误
4. 只读命令不能写 tracked 文件
5. build exit 0 必须等于"构建产物完整且可用"

## 4. Step 1：清理格式闸门

### 目标

`deno task fmt:check` 必须通过（exit 0）。

### 背景问题

当前 16 个文件未格式化，主要是：

- Markdown 文件中的表格对齐（`deno fmt` 会自动对齐 markdown 表格列宽）
- JSON 文件缺少末尾换行
- TypeScript 文件中的引号风格不一致（双引号→单引号）
- Deno fmt 自身有 Rust panic bug（但 exit code 1 是真实的格式检查失败，不是 panic 导致的）

### 涉及文件

根据实际 fmt:check 输出，涉及文件包括：

- `hub-submission.json`
- `docs/adr/0034-hermetic-hub-snapshots.md`
- `docs/adr/README.md`
- `docs/conversation/www-navigation-decision.md`
- `README.md`
- `README.en.md`
- `docs/changelog/v0.19.0.md`
- `docs/sop/audit-remediation-20260518.md`
- `packages/hub/src/cli/dsd-report-gate.ts`
- `packages/hub/src/snapshot-playwright.ts`

### 执行动作

1. **运行 fmt:check 记录当前状态**：
   ```bash
   deno task fmt:check 2>&1 | tee /tmp/fmt-before.log
   ```

2. **自动格式化全部文件**：
   ```bash
   deno task fmt
   ```
   > ⚠️ `deno fmt` 只做格式化，不做语义修改。对 markdown 表格对齐、引号风格、末尾换行的修改不会改变代码行为。

3. **检查格式化引入的变更**：
   ```bash
   git diff --stat
   ```
   - 预期变更类型：表格对齐、引号→单引号、末尾换行、缩进调整
   - **不应出现**：逻辑变更、注释内容变更、删除代码

4. **特别检查 `hub-submission.json`**：
   ```bash
   git diff hub-submission.json
   ```
   - 如果只有末尾换行变更，可以提交
   - 如果 `submittedAt` 变了：`git checkout -- hub-submission.json` 恢复

5. **确认 fmt:check 通过**：
   ```bash
   deno task fmt:check
   # 预期: exit 0（可能有 Rust panic 输出，但无 "not formatted" 行）
   ```

6. **提交格式化变更**：
   ```bash
   git add -A
   git diff --cached --stat  # 确认只有格式化变更
   git commit -m "style: deno fmt — align tables, quotes, trailing newlines"
   ```

### 验收命令

```bash
deno task fmt:check
```

### 通过标准

- `deno task fmt:check` exit 0
- 无 "not formatted files" 报告
- Deno Rust panic 不影响通过标准（panic 是 Deno bug，不是项目问题）

### 失败时如何处理

- 如果 `deno fmt` 后仍有文件 not formatted：可能是 deno fmt 不支持的文件类型，手动修复
- 如果格式化引入了非格式变更：`git diff` 审查后决定是否保留
- 如果 `hub-submission.json` 的 `submittedAt` 被修改：恢复并考虑加入 `.gitignore`

### 是否会污染工作区

否。`deno fmt` 修改文件但不引入功能变更。提交前用 `git diff --stat` 确认变更范围。

## 5. Step 2：修复或隔离 E2E 失败

### 目标

E2E 测试全绿。如果 Windows 本地无法全绿，必须有 CI（Linux）证明。

### 背景问题

审计报告显示 E2E 1/92 失败（`accessibility-performance.spec.ts:75`，Windows `os error 6` 句柄无效）。这是 Windows-specific 的 Playwright/子进程句柄回收问题。

### 涉及文件

- `www/e2e/playwright.config.ts`
- `www/e2e/accessibility-performance.spec.ts`

### 执行动作

1. **清理旧测试产物**：
   ```bash
   rm -rf www/test-results/
   ```

2. **确保 dev server 不占用端口**：
   ```bash
   # 检查 4174 端口
   netstat -ano | grep 4174 || echo "Port 4174 is free"
   ```

3. **运行 E2E 测试**：
   ```bash
   deno task test:e2e
   ```

4. **分析结果**：
   - 如果 92/92 通过：✅ 完成
   - 如果失败是 Windows `os error 6`：
     a. 确认 `www/e2e/playwright.config.ts` 中 `retries` 已设为 1（当前状态）
     b. 再跑一次，确认是否偶发
     c. 如果持续失败：检查是否可以修改 teardown 逻辑或使用 `fullyParallel: false`

5. **如果 Windows 持续失败**：
   - 在 `www/e2e/playwright.config.ts` 中增加注释：
     ```typescript
     // NOTE: E2E on Windows may encounter os error 6 (handle invalid) during
     // worker teardown. This is a known Windows/Playwright interaction issue,
     // not a product bug. CI runs on Ubuntu pass reliably.
     ```
   - 确保 CI（Ubuntu）能全绿：推送后检查 GitHub Actions
   - 在 release notes 中标注：
     > E2E: 92/92 pass on CI (Ubuntu). Windows local may show 1 transient failure
     > (os error 6, Playwright handle) — not a product defect.

6. **如果失败不是 os error 6**：
   - 这是新问题，需要定位根因
   - 不能简单扩大 timeout 掩盖
   - 可能需要标记为 P0 阻塞

### 验收命令

```bash
deno task test:e2e
```

### 通过标准

- **理想**：92/92 pass，0 failed
- **可接受**：91/92 pass，失败仅为 Windows os error 6，且 CI（Ubuntu）92/92 pass

### 失败时如何处理

- 非 os error 6 的失败：P0 阻塞，必须修复
- os error 6 但 CI 也失败：P0 阻塞，说明不是 Windows-specific
- os error 6 且 CI 通过：软阻塞，记录到 Known Issues

### 是否会污染工作区

- `www/test-results/` 会被创建（已在 `.gitignore` 中）
- 不会写 tracked 文件

## 6. Step 3：修复 build 后处理失败语义

### 目标

`deno task build` 不仅 exit 0，而且不包含隐藏的后处理失败。

### 背景问题

审计报告显示 build 日志曾出现 client script 注入 ENOENT 但 exit 0。当前 ENOENT 已消失（可能在前 SOP 修复过程中被间接解决），但 build 仍有以下问题：

1. **6 个 `IMPORT_IS_UNDEFINED` warning**：`@lessjs/ui` 组件的 `default` import 是 `undefined`
2. **Shoelace SSR 渲染错误**：`this.host.querySelector is not a function` 等日志
3. **build 代码中 `catch` 块将客户端脚本注入失败降级为 warn**（`build.ts:177-178`）

### 涉及文件

- `packages/adapter-vite/src/build.ts`（第 177 行：`catch (error) { log.warn(...) }`）
- `packages/adapter-vite/src/ssg-postprocess.ts`
- `packages/ui/src/*.ts`（`default` export 缺失导致 warning）

### 执行动作

1. **确认当前 build 状态**：
   ```bash
   deno task build 2>&1 | tee /tmp/build-output.log
   grep -i "ENOENT\|error\|fail\|warn" /tmp/build-output.log
   ```

2. **分类 build 输出中的问题**：

   | 类型                 | 示例                                         | 严重度 | 是否阻塞                 |
   | -------------------- | -------------------------------------------- | ------ | ------------------------ |
   | ENOENT               | client script 注入目标文件不存在             | P1     | 当前已不出现             |
   | IMPORT_IS_UNDEFINED  | `default` import undefined from `@lessjs/ui` | P2     | 不阻塞                   |
   | Shoelace SSR failure | `this.host.querySelector is not a function`  | P2     | 不阻塞（第三方已知问题） |
   | Chunk size warning   | Total JS exceeds 200 KB                      | P3     | 不阻塞                   |

3. **修复 `build.ts` 中的 catch 块语义**（**此步骤涉及源码修改，需用户确认**）：
   - 当前：client script 注入失败 → `log.warn()`
   - 修改为：client script 注入失败 → `log.error()` + exit 1
   - 但在修改前需确认：这个 catch 是否还覆盖其他合理场景
   - **如果不修改源码**：在 release notes 中标注"build 可能 warn 但不影响产物"

4. **关于 `IMPORT_IS_UNDEFINED` warning**：
   - 这是 Vite 对 `@lessjs/ui` 组件的 `default` export 的 warning
   - 不影响运行时（WC 通过 `customElements.define()` 注册，不走 default export）
   - **v0.19.0 内不修复**，记录为 known issue
   - v0.19.x 可以在各组件文件中添加 `export default class` 或改用 named export

5. **验证 build 产物**：
   ```bash
   deno task build
   # 检查关键页面存在
   ls www/dist/index.html
   ls www/dist/en/index.html
   ls www/dist/registry/index.html
   # 检查 client script 被注入
   grep -l "client/islands/client.js" www/dist/index.html
   ```

### 验收命令

```bash
deno task build
ls www/dist/index.html www/dist/en/index.html www/dist/registry/index.html
```

### 通过标准

- `deno task build` exit 0
- 关键页面 HTML 文件存在
- client script 被注入到 HTML
- 无 ENOENT 错误
- Shoelace SSR 错误已分类为 known（见 Step 4）

### 失败时如何处理

- 如果 ENOENT 重新出现：P0 阻塞，必须修复 ssg-postprocess 的文件查找逻辑
- 如果关键页面缺失：P0 阻塞
- 如果 client script 未注入：P1 阻塞，页面将无交互能力

### 是否会污染工作区

`deno task build` 会写 `www/dist/`（已在 `.gitignore` 中）。不写 tracked 文件。

## 7. Step 4：收紧 DSD report gate

### 目标

`dsd:check-report` 不再以 Infinity 阈值放行，v0.19.0 有明确的可接受标准。

### 背景问题

当前 `dsd-report-gate.ts` 的 non-recoverable 阈值是 `Infinity`（报告模式）。72 个错误全部来自 Shoelace 组件，其中 6 个 `sl-input` instantiate failure 是 non-recoverable。gate 通过只能说明"错误已被分类"，不能说明"DSD 输出健康"。

### 涉及文件

- `packages/hub/src/cli/dsd-report-gate.ts`
- `www/dist/dsd-report.json`（build 产物）

### 执行动作

1. **理解当前 gate 逻辑**：
   ```bash
   cat packages/hub/src/cli/dsd-report-gate.ts | head -80
   ```

2. **确认 72 个错误的分类**：
   ```bash
   deno task build
   deno task dsd:check-report
   ```
   预期输出：
   - 5 种 known error patterns（51+9+6+3+3 = 72）
   - 0 个 unknown errors
   - Gate: passed (threshold: non-recoverable ≤ Infinity)

3. **v0.19.0 收紧方案**：

   **不改代码，改文档标注**（因为改阈值需要修改源码，而用户要求不擅自修改业务代码）：

   a. 在 `docs/status/STATUS.md` 的 Known Issues 中明确：
   ```
   - dsd-report: 72 errors (6 non-recoverable, 66 recoverable), all from Shoelace.
     Gate currently passes at threshold Infinity (report-only mode).
     v0.19.x will tighten to non-recoverable ≤ 6 (current count).
     v0.20 will tighten to ≤ 10 total.
     v0.21 will target 0 unknown errors.
   ```

   b. 在 `dsd-report-gate.ts` 顶部注释中明确当前阈值是 report-only：
   ```typescript
   // NOTE: Current threshold is Infinity (report-only mode).
   // This gate classifies errors but does not fail the build.
   // Tightening schedule:
   //   v0.19.x → non-recoverable ≤ 6 (no new non-recoverable errors)
   //   v0.20   → total errors ≤ 10
   //   v0.21   → 0 unknown errors
   ```

   c. **v0.19.0 最低可接受标准**：
   - ✅ 无 **unknown** error patterns（所有 72 个都已被 5 种 known pattern 匹配）
   - ✅ non-recoverable errors 全部来自 Shoelace `sl-input`（6 个）
   - ✅ 不允许新增 unknown error pattern
   - ❌ 不要求 total errors = 0（这需要 Shoelace 组件改为 SSR-safe，超出 v0.19.0 范围）

4. **如果用户允许修改源码**，建议将阈值从 `Infinity` 改为 `7`（当前 6 + 1 容差）：
   ```typescript
   const MAX_NON_RECOVERABLE = 7; // 当前 6，容差 1
   ```
   这样新增任何 non-recoverable error 都会 fail gate。

### 验收命令

```bash
deno task build
deno task dsd:check-report
```

## 15. 失败处理策略

### P0 阻塞（必须修复才能发布）

| 失败场景                      | 处理方式                                            |
| ----------------------------- | --------------------------------------------------- |
| `fmt:check` 不通过            | `deno fmt` 后重试；如果仍失败，定位具体文件手动修复 |
| E2E 非 Windows-specific 失败  | 定位根因，修复后重跑                                |
| build 产物缺失关键页面        | 排查 build pipeline                                 |
| DSD report 出现 unknown error | 分析错误来源，要么修复要么加入 known patterns       |
| `hub:check-index` 污染工作区  | 代码有 bug，必须修复                                |

### P1 软阻塞（可记录风险后发布）

| 失败场景                                     | 处理方式                                |
| -------------------------------------------- | --------------------------------------- |
| E2E Windows os error 6 且 CI 通过            | 记录 Known Issue，在 release notes 标注 |
| build 有 `IMPORT_IS_UNDEFINED` warning       | 记录 Known Issue                        |
| DSD report non-recoverable = 6（全部 known） | 记录到 release notes                    |
| `hub-submission.json` 漂移                   | 恢复文件，考虑 `.gitignore`             |

### P2 可延后

| 失败场景                 | 处理方式                         |
| ------------------------ | -------------------------------- |
| CI hub-scan job 失败     | v0.19.x 修复，release notes 标注 |
| ADR-0034 未进入 Accepted | 保持 Proposed 状态               |

### 通用回退策略

如果某个 Step 多次失败无法修复：

1. 记录失败现象和已尝试的修复
2. 评估是否可以降级为软阻塞或可延后
3. 如果不能降级，暂停 release，创建 issue 追踪
