# SOP: v0.18.3 - DOM Simulation Experiment

> Purpose: evaluate whether selected browser-dependent Web Components can be
> rendered through an isolated DOM simulation path without weakening the core
> admission model.
>
> Release type: **experimental patch (opt-in only)**.

## Entry Criteria

- v0.17.4 prevents accidental SSR imports.
- v0.18.1 validation can mark packages as `experimental-dom`.
- v0.18.2 install flow can preserve client-only fallback.
- No public docs claim universal third-party SSR.

## Experiment Rules

- Disabled by default.
- Explicit opt-in per package or component.
- Timeout-bound.
- Isolated from normal SSR rendering.
- Always reported in `dsd-report.json`.
- Failure degrades to client-only when policy allows it.

## Step-by-Step Execution

### Step 1: Define Config

Example:

```ts
lessjs({
  ssr: {
    domSimulation: 'off' | 'explicit',
    domSimulationTimeoutMs: 500,
  },
});
```

Default must be `"off"`.

### Step 2: Add Experimental Renderer

Create an isolated renderer path. It must not replace the normal adapter
protocol.

The renderer must:

- create a fresh DOM environment or reset a per-page environment
- define only admitted tags
- enforce timeout
- collect errors and warnings
- serialize only safe HTML/DSD output
- clean up global state

### Step 3: Add Compatibility Fixtures

Fixtures should cover:

- simple browser-dependent component that can serialize
- component that times out
- component that uses unsupported browser API
- component that mutates global state
- component that should remain client-only

### Step 4: Report Results

Report fields:

- `strategy: "experimental-dom"`
- render time
- timeout state
- serialized byte size
- fallback decision
- errors and warnings

## Verification

- [ ] default build does not use DOM simulation
- [ ] explicit fixture can render through experimental path
- [ ] timeout fixture fails predictably
- [ ] unsupported API fixture falls back or fails with clear reason
- [ ] `dsd-report.json` records strategy and result
- [ ] no package is silently upgraded from client-only to SSR
- [ ] `deno task fmt && deno task lint && deno task typecheck`
- [ ] `deno task test`
- [ ] `deno task build`

## Exit Criteria

- Decision is made: continue, limit, or drop DOM simulation.
- Experimental behavior is documented and isolated.
- v1.0 promises remain unchanged: deterministic outcomes, not universal SSR.

## Non-Goals

- Do not make DOM simulation default.
- Do not call this production-ready.
- Do not promise browser-equivalent behavior.
- Do not use DOM simulation to bypass manifest validation.
