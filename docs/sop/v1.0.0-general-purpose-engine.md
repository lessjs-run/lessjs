# SOP: v1.0.0 - Stable General-Purpose WC Engine

> Purpose: freeze the public engine contracts after the renderer, manifest,
> validation, client-only fallback, and registry flows are proven.
>
> Release type: **major (API freeze, stability guarantee)**.

## Goal

1. **Stable engine API**: no breaking changes without a new major version.
2. **Stable package admission**: every package gets SSR/SSG, client-only, or
   rejected with an actionable reason.
3. **Stable manifest/report schemas**: compatibility and render reports are
   safe for tooling and Hub ingestion.
4. **Stable install flow**: `less add` updates projects only after validation.

## Entry Criteria

- v0.17.4 SSR admission planner is stable.
- v0.18 manifest validation and add/install flows are stable.
- v0.19 Hub records and quality gates consume the same report schemas.
- Public docs no longer contain "any CEM package automatically SSRs" claims.

## v1.0 Promise

LessJS v1.0 should promise this:

> A validated Web Component package can be installed and built with a
> deterministic outcome: SSR/SSG when supported, client-only fallback when
> browser-dependent, or a clear validation error before build output.

LessJS v1.0 should not promise this:

> Every CEM package or npm package automatically SSR-renders.

## Scope

| - | ---------------------------- | ---------------------------------- | ---------- |
| 1 | API audit + stability labels | All public packages                | 2 sessions |
| 2 | Schema freeze                | `packages/core/src/types.ts`, docs | 2 sessions |
| 3 | Plugin interface             | `packages/core/`                   | 3 sessions |
| 4 | Compatibility fixture suite  | `packages/*/__tests__/fixtures/`   | 2 sessions |
| 5 | Migration guide              | `docs/`, `www/`                    | 2 sessions |
| 6 | Hub integration contract     | `packages/hub/`, `docs/`           | 1 session  |

## Stability Requirements

- renderer protocol is documented and versioned
- manifest schema is documented and versioned
- `dsd-report.json` schema is documented and versioned
- `less validate-manifest` output is stable enough for CI
- `less add` has dry-run and rollback-safe behavior
- client-only fallback behavior is stable
- experimental DOM simulation is clearly marked experimental if present
- all public APIs are marked stable, experimental, or internal

## Step-by-Step Execution

1. Audit every exported API and mark it stable, experimental, or internal.
2. Freeze manifest, compatibility, and `dsd-report.json` schemas.
3. Freeze renderer protocol and plugin interface.
4. Run migration tests from representative v0.x projects.
5. Verify SSR-capable, client-only, rejected, and experimental fixtures.
6. Publish v1.0 migration guide.
7. Tag v1.0 only after all release gates and docs gates pass.

## Verification

- [ ] `less init && less add <validated-package> && deno task build` succeeds
      without manual project patching
- [ ] SSR-capable fixture renders DSD output
- [ ] client-only fixture skips SSR and hydrates/registers on the client
- [ ] rejected fixture fails before SSR bundle generation
- [ ] Hub can ingest validation and build reports from a v1.0 project
- [ ] all exported APIs have stability labels
- [ ] v0.x migration guide is published
- [ ] `deno task fmt && deno task lint && deno task typecheck`
- [ ] `deno task test`
- [ ] `deno task build`
- [ ] `deno task test:e2e`

## Key Risk

API freeze is a product commitment. If compatibility tiers, report schemas, or
install behavior are still changing weekly, v1.0 should wait. It is better to
ship one more v0.x release than to freeze an unstable package contract.
